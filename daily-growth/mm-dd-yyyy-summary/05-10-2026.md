## Phase A: Session ↔ editor-host bridge — complete

Added two methods to `StackerSession` (`src/stacker/session.rs`):

- **`sync_to_view(&self, view: &mut BufferView)`** — mirrors the
  session's flat-char `StackerSelection` into the view's `(line, col)`
  cursor + anchor via `Buffer::char_to_pos`. Resets `desired_col` and
  clears `extra_cursors` so transient view state from a prior buffer
  doesn't leak. Marked range is intentionally not propagated — the
  editor view does not yet model IME composition; the prose render
  path will consult `marked_range()` separately for underline drawing
  in Phase C/D.
- **`sync_from_view(&mut self, view: &BufferView)`** — converts the
  view's cursor + anchor back to a `StackerSelection` via
  `Buffer::pos_to_char`. Only writes the session if the selection
  actually changed, so unrelated frame work doesn't churn selection
  history.

A2 ownership decision baked in: `StackerUiState.prose_view` is already
constructed via `BufferView::default()`, which produces a
prose-appropriate view (no `git_gutter`, no `lang_id`, no syntax
tree). No state migration needed.

`buffer_mut()` remains the sole mutation entry point used by
`StackerInputEngine` from the input client. Editor-view keyboard
handling is already short-circuited by `prose_mode = true` (when we
flip it in Phase B), so there is no double-mutation risk after Phase C.

### Performance: build profile

User reported `cargo run` was nearly unusable — "one word per minute"
typing in the stacker. Diagnosed: not a code regression, just
unoptimized debug-build dependencies. wgpu / glyphon / egui / ropey
are 10–100× slower at `opt-level = 0`.

**`Cargo.toml`** changes:

- `[profile.dev.package."*"] opt-level = 3` — dependencies compile
  with full optimisations even in dev. First rebuild took **8m 25s**

### Stacker dictation: routing fixes

User reported dictation wasn't reaching the input client at all
("not working") despite Phase 4e shipping. Audit found two real
defects in `stacker_input_client.rs`:

1. **Missing `isFlipped` override.** The view inherits from `NSView`
   (Y-up) but its parent contentView is flipped (Y-down) via winit.
   The math in `firstRectForCharacterRange:` and
   `characterIndexForPoint:` already produced Y-down values, so they
   were being interpreted in the wrong coordinate system. macOS
   dictation queries the rect to anchor its overlay; an
   upside-down rect can cause the system to refuse to start dictation
   against the view.
2. **No formal `NSTextInputClient` protocol conformance.** The class
   implemented every required method but the runtime reported it as
   non-conforming. AppKit's input system uses `conformsToProtocol:`
   when deciding whether to route dictation/IME to a view — without
   formal conformance, dictation silently skips the view.

Fix progression:

- **Attempt 1**: added `isFlipped` method returning `true` plus
  `unsafe impl NSTextInputClient for LlnzyStackerInputClient {}`
  inside `define_class!`. Compiled, but panicked at startup:
  `must implement required protocol method
  -[NSTextInputClient attributedSubstringForProposedRange:actualRange:]`.
  Cause: the protocol's Rust trait declares
  `Retained<NSAttributedString>` / `Retained<NSArray<…>>` returns
  for several methods, and objc2 0.6's `define_class!` macro can't
  bridge those (`Retained<T>` doesn't implement `EncodeReturn`).
- **Attempt 2**: tried matching the trait signatures —
  `validAttributesForMarkedText` returning
  `Retained<NSArray<NSString>>`, `attributedSubstringForProposedRange:`
  returning `Option<Retained<NSAttributedString>>`. Yielded 10
  `EncodeReturn` / `Encode` / `OptionEncode` errors. Confirmed the
  macro really doesn't support `Retained<T>` returns from
  registered methods.
- **Attempt 3 (final)**: kept raw-pointer signatures exactly as they
  shipped in Phase 4e, removed the formal `unsafe impl`, and added
  runtime protocol registration:

  ```rust
  fn ensure_text_input_client_protocol_registered() {
      static REGISTERED: OnceLock<()> = OnceLock::new();
      REGISTERED.get_or_init(|| {
          extern "C" {
              fn class_addProtocol(
                  cls: *mut AnyClass,
                  protocol: *const AnyProtocol,
              ) -> bool;
          }
          let proto = AnyProtocol::get(c"NSTextInputClient")
              .expect("NSTextInputClient protocol must be registered by AppKit");
          let cls: *const AnyClass = LlnzyStackerInputClient::class();
          unsafe {
              class_addProtocol(cls as *mut AnyClass, proto as *const AnyProtocol);
          }
      });
  }
  ```

  Called once from `StackerInputClient::new`. Bypasses the macro's
  strict type-encoding check; AppKit's input system only ever calls
  `conformsToProtocol:` (selector-matching) at runtime, which now
  returns YES.

Final state of `stacker_input_client.rs`:

- `isFlipped = true` method registered (line 135-145).
- All protocol-method signatures unchanged from Phase 4e.
- `ensure_text_input_client_protocol_registered()` called once at
  `StackerInputClient::new` (line ~410).
- No `unsafe impl NSTextInputClient` declaration (incompatible with
  objc2 0.6's macro).

Builds clean, runs without panic. **Live verification of dictation

### What we fixed today

- **Character doubling on every keystroke.** Trace showed two writes
  per key: winit keyboard path (`handler.rs:520-533` →
  `append_text_to_stacker_editor`) **and** AppKit's
  `interpretKeyEvents:` → `insertText:` on the subview. Both fired
  because AppKit routes synthetic CGEvents through interpretKeyEvents
  on winit's view, which propagates `insertText:` to our subview's
  input context. Fix: gated the winit keyboard / IME paths into
  Stacker behind `#[cfg(not(target_os = "macos"))]`. macOS now uses
  `NSTextInputClient` as the sole text input surface.
- **Backspace doing nothing visible.** Trace showed the delete fired,
  then AppKit *also* delivered the Backspace as `insertText: "\u{8}"`
  to our subview, which re-inserted the control char and undid the
  visible delete. Fix: control-char filter at the top of
  `apply_stacker_input_client_insert_text` (`main.rs:256`) drops any
  `insertText:` payload that's entirely control chars (excluding
  `\n` / `\t`).
- **Wispr's "field not detected" regression.** After the doubling
  fix, Wispr refused to deliver at all — when user switched to a
  shell tab the queued transcription immediately flushed there.
  Diagnosis: Wispr only writes to AX-aware fields and our subview
  had no AX surface. Fix: added the full hand-rolled
  `NSAccessibility` text protocol on `LlnzyStackerInputClient`
  (`accessibilityRole=AXTextArea`, `accessibilityValue`,
  `setAccessibilityValue:`, selected text, selected range, number of
  characters). Wispr immediately started delivering.

### What we couldn't crack

- **Wispr's ~10s delivery latency in Stacker.** Wispr is instant in
  the terminal tab of this same app and in every other app on the
  system. Stacker is the only slow target.

We systematically tested and ruled out every cheap hypothesis:

| Tested | Effect on latency |
|---|---|
| Debug → release build | None |
| `AXTextArea` vs `AXTextField` role | None |
| Legacy single-arg `insertText:` (cmux fix) | Never called by Wispr |
| `isEditable`, `attributedString`, `windowLevel` | None |
| `AXValueChanged` + `AXSelectedTextChanged` notifications | Marginal/noise |
| `AXFocusedUIElementChanged` notification | Marginal/noise |

Conclusive diagnostic: with `respondsToSelector:` probe tracing in
place we logged every selector Wispr / AppKit asked about. After
adding the most promising candidates (`isEditable`, `attributedString`,
`windowLevel`), the probes for those went away — and the latency
**did not move**. Wispr is not gating on any individual AX/responder
method we can reasonably implement.

The remaining probes were either TextKit-specific (`textStorage`,
`layoutManager`, `textLayoutManager` — we must NOT implement these,
they'd advertise us as a real `NSTextView` and Wispr would call
TextKit methods on returned objects that don't exist), private
Apple SPI (`_isNSTextInputContextiOSMacClient`,
`_dynamicContextEvaluation:`), or low-value (vertical-text glyph
methods).

### Option B spike — AccessKit integration

Per the refinements roadmap, kicked off Phase 1 of the
`accesskit_macos` integration as the chosen Option B path.
Outcome: **AccessKit is the wrong tool for this problem.**

- Added `accesskit = "0.24"` and `accesskit_macos = "0.26"` to
  `Cargo.toml`. Note: latest accesskit_macos still targets
  `objc2 0.5` / `objc2-app-kit 0.2` while we're on `0.6` / `0.3`.
  Cargo allows both versions to coexist; `SubclassingAdapter::new`
  takes `*mut c_void` so the boundary handoff is clean.
- Wrote `src/stacker_accesskit.rs` with `ActivationHandler` /
  `ActionHandler` impls and a `build_tree_update` function that
  publishes a minimal `MultilineTextInput` node mirroring
  `StackerSession` state.
- Wired `SubclassingAdapter` into `StackerInputClient::new()`,
  pushed `TreeUpdate`s from `set_state` on every change, and
  forwarded focus state through `update_view_focus_state`.
- **Spike result with hand-rolled AX still in place:** Wispr's
  latency was unchanged. Trace showed Wispr was hitting our
  hand-rolled `accessibilityRole` / `accessibilityValue` / etc.
  directly — AppKit never descends into `accessibilityChildren`
  (where AccessKit injects its tree) when the parent view already
  answers AX queries.
- **Spike result after removing hand-rolled AX methods:** Wispr's
  latency was *still* unchanged. The `ax_*_query` traces went
  silent (no methods left to call) but the 10s wait persisted.
  AppKit / Wispr is not descending into `accessibilityChildren` at
  all. AccessKit's tree is published but invisible to Wispr's
  delivery logic.

Diagnosis: Wispr's classification keys on something outside the AX
protocol — almost certainly the responder's *class name*
(`LlnzyStackerInputClient` is unknown to Wispr's allowlist) and/or
the absence of a TextKit object graph. No amount of AX surface
work can flip that classification because Wispr isn't reading our
AX.

### First-responder diagnostic — and the focus thrash

Added a per-frame `diag_log_first_responder()` to
`StackerInputClient` that queries `[window firstResponder]`,
extracts the class name (with `(== spike NSTextView)` /
`(== LlnzyStackerInputClient)` annotations for clarity), and
emits a stderr line whenever the identity changes. Called from
the start of `set_state`, which runs every frame. Dedup'd via a
`static LAST: OnceLock<Mutex<String>>` so quiet frames stay
quiet.

Result on the first run with the diagnostic:

```
<init> -> NSKVONotifying_WinitView
WinitView -> NSTextView (spike NSTextView)
NSTextView -> LlnzyStackerInputClient
LlnzyStackerInputClient -> NSTextView
NSTextView -> LlnzyStackerInputClient
[…dozens of flips…]
```

The first responder **thrashed** between the spike `NSTextView`
and our legacy `LlnzyStackerInputClient` constantly during
dictation. Both subviews overlap the same frame; AppKit's hit-test
on incoming events kept landing one of them, then our
`ensure_focused()` (called per-frame from
`sync_stacker_input_client`) flipped it back to the spike via
`makeFirstResponder:`. Wispr's classification was hitting a
moving target — the Phase 1 result was invalid.

### Spike-active flag — neutralizing the legacy subview

Added a `static SPIKE_ACTIVE: AtomicBool` set in
`StackerInputClient::new()` when the env var is on. Modified
`LlnzyStackerInputClient::acceptsFirstResponder` to return
`!SPIKE_ACTIVE && ACTIVE` instead of just `ACTIVE`. With the
spike enabled, AppKit now *refuses* to make
`LlnzyStackerInputClient` first responder — the spike `NSTextView`
is the only valid target. (Couldn't use `if ... return false`
inside the `define_class!` method because objc2 0.6's macro
expects the return type to coerce as `objc2::runtime::Bool`,
and `return false;` defeats that coercion — wrote it as a single
expression instead.)

### Clean Phase 1 result

Re-ran with the spike + the new acceptsFirstResponder gate.
Diagnostic showed:

```
<init> -> NSKVONotifying_WinitView
WinitView -> NSTextView (spike NSTextView)
[stable for the whole session — no thrash]
```

Spike was the sole first responder for the entire Wispr
dictation. User report: latency unchanged, ~8-10s. Same as
before. **Phase 1 gate definitively failed.**

Significance: this is *the* gold-standard test. A stock
`NSTextView` — the exact class TextEdit uses, with intrinsic
full `NSAccessibility` text surface and intrinsic
`NSTextInputClient` conformance — held focus alone, and Wispr
still took ~10s. We now know with certainty that **no
view-level architecture change inside our app can solve this.**

### The TextEdit baseline

| Setup | Wispr latency |

| llnzy `cargo run` + spike NSTextView (sole responder) | ~10s |

The class identity, AX surface, and protocol conformance are

### Conclusion

Wispr Flow has a per-app classification step (almost certainly an
internal allowlist of popular apps it's already learned to handle
fast; everyone else gets routed through their AI-rewrite pipeline,
which adds ~10s). TextEdit and known apps are on the list. We
aren't. No amount of view-level work on our side moves us onto
that list.

llnzy's input architecture is sound. Every well-behaved native
dictation tool that types into the focused text field works —
Superwhisper proved it. Wispr is the outlier.

  re-added** to `LlnzyStackerInputClient`:
  - `isAccessibilityElement` → `true`

    (handed out as a raw `*mut AnyObject` from the Foundation
    constant — interned, retain/release are no-ops, no
    autorelease balance needed).

    materialises the buffer via the snapshot and returns an
    autoreleased `NSString` (factory convention on
    `stringWithUTF8String:`); setter posts an
    `InsertText { replacement_utf16: Some((0, total_u16)) }`
    to replace the whole buffer.
  - `accessibilitySelectedText` / `setAccessibilitySelectedText:`
    — getter returns the current selection slice; setter posts
    `InsertText` with the selection's UTF-16 range as
    replacement.
  - `accessibilitySelectedTextRange` — returns the sorted
    selection as `NSRange` in UTF-16.
  - `accessibilityNumberOfCharacters` — returns total UTF-16
    char count.
