## Phase 3: Core Editing & Multi-Buffer -- COMPLETE

### Clipboard (Cmd+C/X/V)
- Copy: selection -> clipboard; no selection -> copy whole line
- Cut: selection -> clipboard + delete; no selection -> cut whole line
- Paste: insert at cursor, replacing selection if active
- Two-way clipboard bridge: editor writes `clipboard_out`, main loop reads arboard and supplies `clipboard_in`

### Line Operations
- Cmd+Shift+K: delete entire line
- Cmd+Shift+D: duplicate line below
- Alt+Up/Down: move line up/down (swap with adjacent line)

### Indent/Dedent
- Tab: insert indent at cursor (respects file's detected indent style)
- Tab with selection: indent all selected lines by one level
- Shift+Tab: dedent (selection or single line)

### Auto-Pairing
- Auto-close on open bracket: `()`, `[]`, `{}`, `""`, `''`, backtick pairs
- Skip-over: typing a closing bracket jumps past it if it matches
- Pair deletion: backspace between empty pair deletes both characters
- Context-aware: only auto-pairs when next char is whitespace/closing/EOL

### Smart Enter
- Auto-indent: copies leading whitespace from current line
- Extra indent after `{`, `(`, `[` (detects trailing open bracket before cursor)

### Multi-Buffer Tabs
- Tab bar rendered above editor with file names
- Modified indicator (`*`) on unsaved tabs
- Click tab to switch, middle/right-click to close
- Active tab highlighted in blue, inactive in dark grey
- Per-buffer cursor + scroll state preserved when switching tabs
- "< Files" button returns to directory browser without closing buffers

## Phase 5: Tree-Sitter Syntax Highlighting -- COMPLETE

### `src/editor/syntax.rs` (550 lines, 6 tests)
- `SyntaxEngine` wrapping tree-sitter `Parser` + compiled `Query` per language
- `detect_language(path)` maps file extensions to language IDs
- `parse(lang_id, source)` -- full parse on file open
- `reparse(lang_id, source, old_tree)` -- incremental re-parse after edits (O(edit-size))
- `highlights_for_range(lang_id, tree, source, start_line, end_line)` -- returns colored spans for the visible viewport only
- Byte-range restriction on `QueryCursor` for performance (only queries visible lines)
- `StreamingIterator`-based query match iteration (tree-sitter 0.26 API)

### 16 Highlight Groups
Keyword, Type, Function, Variable, String, Number, Comment, Operator, Punctuation, Constant, Attribute, Tag, Property, Escape, Label, Module -- each mapped to an RGB color (One Dark-inspired palette).

### Supported Languages (11)
| Language | Grammar Crate | Version |
|----------|--------------|---------|
| Rust | `tree-sitter-rust` | 0.24 |
| JavaScript | `tree-sitter-javascript` | 0.25 |
| TypeScript | `tree-sitter-typescript` | 0.23 |
| TSX | `tree-sitter-typescript` | 0.23 |
| Python | `tree-sitter-python` | 0.25 |
| Go | `tree-sitter-go` | 0.25 |
| C | `tree-sitter-c` | 0.24 |
| JSON | `tree-sitter-json` | 0.24 |
| HTML | `tree-sitter-html` | 0.23 |
| CSS | `tree-sitter-css` | 0.25 |
| Bash | `tree-sitter-bash` | 0.25 |

TOML grammar (`tree-sitter-toml` 0.20) is incompatible with tree-sitter 0.26 -- query and detection code is ready, waiting for upstream update.

### Editor Integration
- `EditorState.open()` detects the language from file extension and parses the initial syntax tree on open
- After any edit that changes content length, `view.tree_dirty` is flagged
- Before each render frame, `reparse_active()` incrementally updates the tree using the old tree as a reference
- The editor renderer fetches highlight spans for only the visible line range
- Text rendering batches consecutive characters with the same highlight color into single draw calls
- Files with no recognized grammar render as plain white text (graceful fallback)

### Highlight Query Design
Each language has a compact, hand-written query using tree-sitter's S-expression pattern syntax. Queries target the most visually impactful nodes: keywords, string/number literals, comments, function names, type identifiers, attributes, properties, operators, and punctuation. Capture names map to `HighlightGroup` via prefix matching (e.g. `keyword.return` -> `Keyword`).

The Rust query uses `concat!()` of string literals instead of `r#"..."#` to avoid conflicts with tree-sitter query `"#"` patterns. Other languages use standard raw strings.

## Phase 6: LSP Client Foundation -- COMPLETE

### `src/lsp/transport.rs` (220 lines)
- Content-Length framed JSON-RPC over child process stdio (tokio async)
- Async reader task parses headers, reads body, dispatches messages
- Request/response matching via `AtomicI64` ID counter + `oneshot` channels
- Server notifications forwarded to `mpsc::UnboundedReceiver`
- `EventLoopProxy::send_event(LspMessage)` wakes the main thread on incoming messages

### `src/lsp/client.rs` (385 lines)
- Full `initialize` handshake with client capabilities (text sync, completion, hover, diagnostics, definition, references)
- Document lifecycle: `did_open`, `did_change` (full sync), `did_save`, `did_close`
- LSP requests: `hover`, `definition`, `completion`
- Graceful `shutdown` + `exit` sequence
- URI helpers: `path_to_uri` / `uri_to_path` for file:// conversions

### `src/lsp/registry.rs` (85 lines)
- 11 language server configs: rust-analyzer, typescript-language-server, pyright, gopls, clangd, bash-language-server, vscode-html/css/json-language-server
- `is_available(command)` checks PATH via `which`
- `find_server(lang_id)` returns config only if server is installed

### `src/lsp/mod.rs` (229 lines)
- `LspManager`: owns tokio `Runtime`, manages multiple `LspClient` instances
- `ensure_server(lang_id)`: starts server on first use (blocking init)
- Document sync methods delegate to the right client by lang_id
- `handle_notification`: dispatches `publishDiagnostics` to `HashMap<PathBuf, Vec<FileDiagnostic>>`
- `detect_root(path)`: walks up to find `.git`, `Cargo.toml`, `package.json`, etc.

### Editor Integration
- `EditorViewState.init_lsp(proxy)`: lazily creates the manager from the event loop proxy
- `EditorViewState.open_file(path)`: detects project root, starts server, sends `didOpen`
- Content-length change detection after key handling triggers `didChange`
- Save detection (was_modified -> !is_modified) triggers `didSave`

## Phase 7.1: Diagnostics Rendering -- COMPLETE

- Squiggly underlines rendered as alternating line segments: red (error), yellow (warning), blue (info), grey (hint)

## Architecture Decisions

1. **Ropey over raw String**: O(log n) insertions make the editor viable for large files without recopying the entire document on each keystroke
2. **Per-buffer view state**: Each open buffer retains its own cursor position, scroll offset, language ID, and tree-sitter parse tree -- switching tabs restores exactly where you were with full syntax highlighting
3. **egui painter for rendering**: Started with egui's custom painting API rather than the direct glyphon pipeline. Faster iteration on UI features; can migrate to glyphon later if performance demands it
4. **Clipboard bridge pattern**: Editor can't directly access `arboard` (it's owned by the main `App`), so clipboard flows through `clipboard_out`/`clipboard_in` fields that the main loop mediates
5. **Explorer -> EditorState routing**: Text files open through `EditorState` (multi-buffer tabs with syntax trees); images stay in `explorer.open_file` (single preview). Clean separation
6. **Incremental tree-sitter parsing**: On file open, the full document is parsed. After edits, only the changed regions are re-parsed using `reparse()` with the old tree -- this keeps syntax highlighting responsive even on large files
7. **Viewport-restricted highlight queries**: `QueryCursor::set_byte_range()` restricts tree-sitter queries to only the visible line range, avoiding O(file-size) work on every render frame
8. **Content-length change detection**: Compares `buf.len_chars()` before/after key handling to detect edits for both tree-sitter re-parse and LSP `didChange` -- simple, correct, zero-overhead when idle
9. **Synchronous LSP on tokio runtime**: `LspManager` owns a tokio `Runtime` and uses `block_on()` for LSP calls. This keeps the API synchronous from the editor's perspective while the transport reader runs async. Server notifications arrive via `EventLoopProxy` waking the main thread
10. **Lazy server startup**: Language servers are only spawned when the first file of that language is opened, not at app startup. `ensure_server()` checks PATH availability before attempting to spawn
11. **Project root detection**: Walks up from the opened file to find `.git`, `Cargo.toml`, `package.json`, etc. and sets the LSP workspace root accordingly
