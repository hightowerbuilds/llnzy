Fixed Shell + Sketch joined-tab focus routing.

Completed:

- Updated tab switching sync so the UI immediately knows the active tab is Terminal after the shell pane is clicked.
- This prevents Sketch's terminal-input guard from swallowing the shell click/key path after switching panes.
- Followed up after manual testing showed joined Sketch still blocked tab hotkeys:
  - Global app keybindings now run before Sketch can capture keyboard events.
  - `Command-1`, `Command-2`, `Command-3`, and related tab commands can switch tabs even when Sketch is joined with a shell.
  - Clicking a joined terminal pane clears egui text focus so previously focused Sketch controls cannot keep consuming terminal typing.

  - Joined panes both render every frame, so inactive Sketch was still processing its active text draft.

  - Inactive joined Sketch panes still draw, but no longer process text drafts, shortcuts, paste, canvas pointer actions, or inline cursor input.
- Fixed mouse-based focus switching between joined Sketch and Shell panes:
  - Pane activation now uses the full rendered joined pane bounds instead of only the terminal text grid rectangle.
  - Clicking anywhere in the joined terminal pane clears egui text focus before switching so Sketch controls do not retain keyboard ownership.
- Tried one additional terminal-body event routing adjustment, then walked it back after manual testing showed the problem was still present.
- Reviewed the joined Sketch+Terminal input path and found the main quality issue is responsibility spread, not obvious dead code:
