## Why The WebView Existed

Traced the actual reason through git log and the May 4 build summary:

- The WebView landed in commit `7af1efd` (`Add WebView-backed Stacker editor`).
- The summary on `05-04-2026.md` records the trigger plainly: voice dictation,
  specifically Wispr Flow. macOS dictation, IME, and AutoFill all hook the OS
  text-input protocol (`NSTextInputClient`). egui's painted `TextEdit` does
  not satisfy that protocol.
- A first attempt used a hidden invisible `NSTextView` bridge in
  `src/macos_text_bridge.rs`. It was fragile because dictation needs real
  on-screen geometry to anchor its overlay, and focus regains had races.
- The WebView was a pragmatic workaround. WKWebView is already a fully featured
  `NSTextInputClient` with on-screen geometry, so dictation and Wispr Flow
  delivered cleanly.

## Roadmap For Native Input

Recorded the path forward in
`daily-growth/roadmaps/stacker-native-input-roadmap-05-09-2026.md`:

- **Option A (now):** Replace the WKWebView with a visible `NSTextView` inside
  an `NSScrollView` overlay. Same architectural shape (native subview clipped
  to the Stacker pane), no JS/HTML/Chromium, dictation still routed by AppKit.
- **Option C (follow-up):** Phased plan to fold Stacker into the existing
  editor as a `BufferKind::Prose` buffer rendered through glyphon, with
  `NSTextInputClient` implemented on the winit content view itself.

## Continued — NSTextInputClient Phase 4c–4e (same day, later session)

Picked up from Phase 4b (`LlnzyStackerInputClient` subview shipping with two
stubs: `firstRectForCharacterRange:` returning the whole view frame, and
`characterIndexForPoint:` returning 0). Completed the remaining protocol
implementation and tore out the last remnants of the NSTextView path.

### Phase 4c — `firstRectForCharacterRange:` with real glyph data

The dictation and IME overlay anchors to wherever `firstRectForCharacterRange:`
reports the caret is. The Phase 4b stub returned the entire input view's screen
frame, so the overlay appeared at the top-left of the editor rect rather than
at the actual cursor.

Fix: after each egui paint, `render_stacker_editor` now returns the live
`Arc<egui::Galley>` and its top-left origin alongside the editor rect. This
data flows through `StackerPaintOutput` → `editor_panel` → `stacker_view` →
`StackerUiState.web_editor_galley`. Every frame `sync_stacker_input_client`
calls the new `StackerInputClient::set_galley` to push it into
`ClientSnapshot`.

`firstRectForCharacterRange:` then:
1. Resolves the incoming UTF-16 range to a char index.
2. Calls `galley.pos_from_ccursor` to get the character rect in galley-local
   coordinates.
3. Adds the galley origin to get egui absolute screen coordinates.
4. Subtracts the view's frame origin to get view-local coordinates.
5. Converts to window base coordinates via `convertRect:toView:nil`.
6. Converts to screen coordinates via `convertRectToScreen:`.

Files changed: `editor_paint.rs`, `editor_panel.rs`, `stacker_view.rs`,
`stacker_state.rs`, `tab_content.rs` (×2 call sites), `stacker_input_client.rs`,
`main.rs`. Seven files, zero test regressions.

Commit: `3a18f9c` on `stacker-development`.

### Phase 4d — `characterIndexForPoint:` and `attributedSubstringForProposedRange:`

**`characterIndexForPoint:`** — the inverse of Phase 4c. Converts the incoming
screen point back through window-base → view-local → egui absolute → galley-local,
then calls `galley.cursor_from_pos` to find the nearest character and returns
its UTF-16 code unit offset.

**`attributedSubstringForProposedRange:`** — previously returned `nil`. CJK IMEs
(Japanese, Chinese, Korean) read surrounding document context from this method
to rank composition candidates; returning nil causes repeated characters and
incorrect candidates. Now extracts the UTF-16 slice from `ClientSnapshot.text`,
creates an `NSString` via `stringWithUTF8String:` (null-terminated UTF-8 byte
slice), wraps it in an `NSAttributedString` via `alloc/initWithString:`, and
returns it autoreleased. The `actual_range` out-parameter is filled with the
clamped UTF-16 range actually served.

Commit: `6ea6e81` on `stacker-development`.
