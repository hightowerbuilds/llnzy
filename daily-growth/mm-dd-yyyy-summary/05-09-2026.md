## CLI Subcommand (v2 Step 4)

Landed the headless `llnzy prompt add` entry point used by external agents.

Completed:

- Added `src/stacker/cli.rs` (~600 lines including tests) with:
  - `run_from_env()` — reads argv, stdin, stdout, stderr, and config paths from the process environment.
  - `run<R, W, E>(args, stdin, stdout, stderr, paths)` — pure entry point taking readers and writers, fully testable with `&[u8]` and `Vec<u8>`.
  - Hand-rolled flag parser with no clap dep: `--label` (required), `--category`, `--workspace`, `--source-agent`, `--session`, `--file`.
  - Trust boundary fully enforced:
    - 256 KB body cap on stdin (via `Read::take`) and on `--file` input (via metadata after `canonicalize` to follow symlinks).
    - UTF-8 validation on every body source.
    - Regular-file check on `--file` source after symlink resolution.
    - Label sanitization: control chars stripped, length truncated to 256 chars.
    - Category sanitization: control chars stripped, length truncated to 64 chars.
    - Inbox quota check before write: refuses at 1000 files or 50 MB with exit 3.
  - Exit codes: 0 success, 1 usage error, 2 input error, 3 quota exceeded.
  - Writes via the storage layer: ULID id, `state: pending`, `created` = now, body verbatim.
- Added `pub mod cli` to `src/stacker.rs`.
- Hooked CLI dispatch into `src/main.rs` immediately after `env_logger::init()`. CLI mode `std::process::exit`s before the panic handler, signal setup, or `EventLoop::new()` ever run, so an agent calling the CLI never spins up wgpu or winit.

Validation:

  - Unknown flag exits usage.
  - Empty body (only whitespace) exits input error.
  - Oversize stdin body exits input error.
  - Non-UTF-8 stdin body exits input error.
  - Happy path with all flags writes a pending prompt to inbox.
  - `--file` source reads body from disk.
  - `--file` directory rejected.
  - `--file` missing path rejected.
  - `--file` oversize payload rejected.
  - Long label is truncated to 256 chars and control chars stripped.
  - Label of only control chars rejected.
  - Quota blocks when inbox at file limit.

- Full `cargo build --release` succeeds.

- Replaced the old saved-library read path:

  - First load runs `storage::migrate_legacy_json(...)` before listing `prompts/saved/`.
  - Failed migrations or malformed saved prompt files warn and degrade to the successfully readable library rather than crashing the UI.
- Replaced the old saved-library write path:
  - Added `persist_prompt_library()` / `persist_prompt_library_at()`.
  - New prompts get a ULID on first persist and are written as Markdown prompt records under `saved/`.
  - Edited prompts preserve their original `created` timestamp.
  - Removed prompts move from `saved/` to `archive/` instead of being deleted outright.
  - No-op persists skip rewrites so file mtimes stay stable for future watcher work.
- Updated `ui/stacker_state.rs` to load from `prompts/saved/`, keep a `last_persisted_prompts` snapshot, and persist only through the new storage diff.
- Left `stacker_queue.json` unchanged; queued prompts remain separate runtime state.

Validation added:

- Load with no legacy file and no saved directory returns an empty library.
- First load migrates legacy `stacker.json`, leaves `stacker.json.migrated`, and returns id-backed prompts immediately.
- Repeated load after migration is idempotent.
- Persisting a new prompt writes a Markdown record and assigns the prompt id in memory.
- Persisting an edit rewrites the same id and preserves `created`.
- Persisting a deletion archives the removed prompt.
- Persisting an unchanged library does not rewrite the saved file.

Sequencing note:

- This closes the safety-valve step that the previous section intentionally deferred. The app can now move the curated library off `stacker.json` before inbox rendering lands, without showing the user an empty Stacker on first launch.
