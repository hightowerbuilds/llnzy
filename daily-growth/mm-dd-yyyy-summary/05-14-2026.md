# 05-14-2026 Summary

## Critical review of the codebase

Walked the tree from the perspective of a senior dev who has shipped IDEs and
terminal emulators, looking at what is built, what is missing, and where the
foundations are load-bearing versus fragile.

What is built (and built well):

- A hand-written, tree-sitter-highlighted, rope-backed editor with undo,
  multi-cursor scaffolding, snippet parser, git gutter, project search, and an
  in-house LSP client speaking 11 servers.
- A real PTY terminal on alacritty_terminal with scrollback, OSC handling,
  selection, and a joined-pane background workflow.
- Discipline markers: ~590 tests, zero TODOs in the tree, structured
  `error_log.rs`, panic-hook-to-crash-log in `main.rs`, SIGPIPE ignored,
  `#[profile.dev]` tuned for typing latency.

Structural concerns surfaced:

- Monolithic surfaces at the size cliff (`gpui_editor.rs` 1,350 lines,
  `gpui_workspace.rs` 1,077, `gpui_terminal.rs` 1,002, `gpui_sketch.rs` 1,204,
  `stacker/cli.rs` 1,410). No plugin boundary, no extension API, no scripting
  surface, no IPC contract between surfaces.
- Polling-not-eventing in `gpui_terminal.rs:37` -- `POLL_INTERVAL_MS = 16`
  wakes 60x/sec to drain the PTY whether or not anything happened.
- Fixed `CELL_WIDTH: f32 = 9.0` decoupled from font, font size, and DPI; wide
  characters (CJK, emoji) will desynchronize rows.
- `snippet_support: Some(false)` in `lsp/client/init.rs:35` despite the
  snippet parser already living in `editor/snippet.rs`.
- Vim mode is a stub on the view model with no operator-pending state machine.

Conspicuously missing for a "developer workspace": command palette, fuzzy file
finder, real pane splits, minimap/breadcrumbs/outline, code-folding UI,
debugger/DAP, remote dev, extension surface, project-wide refactor preview UI,
terminal split, IME/preedit handling, accessibility, auto-update,
signing/notarization, and a Linux/Windows CI matrix.

Code-level smells: 393 `unwrap`/`expect`/`panic!` sites, `gpui = "=0.2.2"`
exact-pinned with no upgrade plan, `tree-sitter-toml` commented out, 14 MB
`LLNZY.dmg` checked into git, no fuzzing or property tests, no benchmarks, no
`cargo deny`/`cargo audit` in CI.

The product question: there is no single thesis tying terminal + editor +
sketch + prompt manager + appearances together. Zed has "fast collaborative."
Helix has "modal, no config." Cursor has "AI native." LLNZY currently reads as
"the things Luke uses, in one window" -- a great reason to build it and a
difficult reason to ship it to anyone else.

## Terminal cell metrics + wide-char handling (roadmap item 1)

Replaced the hardcoded `CELL_WIDTH = 9.0` / `LINE_HEIGHT = 20.0` pair in
`gpui_terminal.rs` with a `CellMetrics` value computed each frame from the
actual terminal font and the configured `terminal.line_height` multiplier.

Changes:

- Introduced a `CellMetrics { advance, line_height }` struct, replacing the two
  module constants. The constants are retained only as `FALLBACK_CELL_WIDTH` /
  `FALLBACK_LINE_HEIGHT` for the first frame before the font resolves.
- Added `compute_cell_metrics(window, base_font, font_size, config)` that
  asks `TextSystem::em_advance` for the natural monospace advance of the
  resolved font and combines `font_size * config.line_height` for the
  vertical step.
- Wired the metrics through every geometry site in `gpui_terminal.rs`:
  `point_to_grid`, `bounds_for_range`, `terminal_grid_size`, `cursor_quad`,
  `selection_rects`, `background_rects`, `decoration_rects`,
  `url_decoration_rects`, the scroll-wheel handler, and the per-line paint
  origin.
- Cached the metrics on `TerminalSurface` as `last_metrics` so event-handler
  paths (mouse hit-testing, IME bounds) use the same numbers as the painted
  geometry. The cache updates inside `resize_for_bounds`, which is called
  from `paint` immediately after `prepaint` produces the metrics.
- Updated `gpui_terminal/effects.rs` so `terminal_cursor_effects` takes
  metrics and drops its `CELL_WIDTH` / `LINE_HEIGHT` imports.
- Updated `gpui_terminal/text.rs` so `terminal_row_runs` skips cells flagged
  `WIDE_CHAR_SPACER` or `LEADING_WIDE_CHAR_SPACER`. The wide glyph itself
  sits in the preceding cell and its natural advance carries it across the
  spacer column; previously the spacer was emitted as a space, pushing every
  cell after a CJK character one column to the right.

Tests:

- Added `terminal::tests::wide_char_occupies_two_cells_with_spacer_flag` to
  pin alacritty's wide-char layout (`中` lands at column 0 with
  `WIDE_CHAR`, column 1 carries `WIDE_CHAR_SPACER`, `A` lands at column 2).

## Live smoke test surfaced two deeper structural issues

The first live launch immediately exposed two problems the unit tests could
not catch.

### Cell advance was measured against a different font than the renderer used

Initial fix attempted: call `TextSystem::em_advance(resolve_font(base_font),
font_size)` and use the result as the cell width.

The cursor still drifted ~10 columns past the end of typed input. Reason:
`em_advance` and the actual `shape_text` rendering of the row do not always
resolve to the same font when the requested family is missing. Berkeley Mono
(the previous default) is not installed on this Mac, so `em_advance`
measured one fallback and `shape_text` rendered with another. Two different
advance widths, accumulating drift across the row.

Second fix: replaced `em_advance` with a `shape_line` probe of `"MMMMMMMMMM"`
through the exact same shaping pipeline that paints rows, dividing the total
shaped width by 10. Same code path means same font resolution by
construction. Also disabled `calt` (programming ligatures) on the terminal
font via `gpui::FontFeatures::disable_ligatures()` because sequences like
`=>`, `->`, `==`, and `!=` collapse into single glyphs in many monospace
fonts and break grid alignment regardless of how the cell width is measured.

The probe-based measurement removed about 80% of the drift but the cursor
was still visibly ahead of typed text. Time for the structural fix.

### The alacritty -> GPUI bridge handed whole rows to `shape_text`

Root cause: every row of the terminal was being concatenated into a single
`String` and shaped through `TextSystem::shape_text`, which is GPUI's
proportional text layout path. Even with a "monospace" font and ligatures
disabled, the shaper can apply per-glyph kerning, fallback metrics,
subpixel rounding, and OpenType features beyond `calt` -- and any tiny
per-glyph drift accumulates across an 80-column row. The cursor was anchored
to `col * advance` while the rendered text was anchored to "wherever the
shaper put each glyph." They could not stay aligned in principle, only by
luck. The user separately reported ASCII art and TUI box drawing (Claude
Code, Gemini) rendering with broken corners and misaligned edges -- same
root cause: the row-wide shape call can't preserve a strict per-column
grid.

Structural fix: replaced row-wide shaping with **per-cell shaping**.

- Removed `terminal_row_runs` (returned one `String` + style runs per row).
- Added `terminal_row_glyphs` that returns a `Vec<TerminalCellGlyph>`,
  one entry per visible cell with its grid column, character, and
  styled `TextRun`. Empty space cells with default style are skipped
  entirely (backgrounds are painted via `Terminal::background_rects`).
- In `prepaint`, loop the glyphs and shape each through `shape_line`,
  storing the resulting `ShapedLine` together with its `(row, col)`.
- Changed `TerminalPaintLine` to carry both `row` and `col`, and switched
  from `WrappedLine` to `ShapedLine` (the simpler shape result that
  matches single-glyph shaping).
- In `paint`, anchor every glyph at exactly `(TERMINAL_PADDING + col *
  metrics.advance, TERMINAL_PADDING + row * metrics.line_height)`. The
  grid alignment is now structural: cursor at column K and glyph at
  column K share the same x coordinate by definition, not by shaper
  cooperation.

With per-cell shaping in, alignment was exact -- but the second live test
showed every character sitting in a cell roughly twice its glyph width.
The grid was correct; the cells were too wide.

### The default font family didn't exist on this Mac

`gpui_terminal.rs` set `.font_family("Berkeley Mono")` on the wrapping div.
Berkeley Mono is a commercial font and was not installed. GPUI resolved the
family through its fallback stack to SF Pro -- a proportional font. The
probe `"MMMMMMMMMM"` measured the width of ten capital M's in SF Pro
(generous), so the cell width was too wide for the average rendered char,
producing visible inter-character gaps. The alignment was perfect; the
spacing looked wrong.

Fix: changed the default `.font_family("Berkeley Mono")` to
`.font_family("Menlo")`. Menlo is a true monospace and ships with every
macOS install, so font resolution does not fall back. Probe and renderer
both produce identical, consistent advances. Users can still override via
`terminal.font_family` in config; Berkeley Mono will work for them if it
is installed.

## Event-driven PTY render loop (roadmap item 2)

Replaced the 60 Hz poll loop on the terminal render task with an event-
driven path. The PTY reader thread was already blocking on `reader.read`
at the OS level, so the wasted work lived entirely in the GPUI side: a
16 ms timer waking the task every frame just to call `try_read` on an
empty channel.

Changes:

- `src/pty.rs` -- added `wakeup: Arc<tokio::sync::Notify>` to `Pty`. The
  reader thread now calls `notify_one()` after every read outcome (data,
  EOF, error path) so the render task wakes on real events and never
  misses a disconnect.
- `src/pty.rs` -- added `Pty::wakeup_handle()` that clones the notifier
  out for consumers.
- `src/session.rs` -- forwarded the notifier with `Session::wakeup_handle`.
- `src/gpui_terminal.rs` -- replaced `start_poll_task` (16 ms timer) with
  `start_event_task`. Each iteration:
  1. Re-fetches the live session's wakeup handle via `surface.update`.
     The re-fetch is required because Cmd-R restart swaps in a fresh
     `Pty` with a fresh notifier.
  2. Awaits `Notify::notified` with a 500 ms safety-net timeout via
     `gpui::FutureExt::with_timeout`. The timeout exists only to cover
     the rare window where the session is replaced between fetch and
     await; under normal operation the task sleeps until real output.
  3. Calls `process_output` on the surface. Returns `cx.notify()` so the
     repaint only fires when data actually changed.
- Removed the old `POLL_INTERVAL_MS = 16` constant; the surviving
  `IDLE_FALLBACK_MS = 500` is just a recovery heartbeat.

Idle CPU sampled from `top -stats pid,command,cpu -pid <workspace>`
dropped to 0.0% from the prior baseline that kept the task ticking at
60 Hz regardless of activity. Live test reported as "feels excellent."

Verification:


## Multi-explorer tabs feature (separate roadmap)

In parallel with the terminal-side work, scoped and built a new feature:
multiple file explorers as workspace tabs, mirroring how terminals already
work. Roadmap saved at `daily-growth/roadmaps/future/explorer-tabs.md`,
implemented end-to-end in five phases on `main` alongside the other
agent's terminal changes.

User-visible behaviour:

- New **Explorer** button in the footer between Home and Terminal.
- First click pops the sidebar's explorer out as Explorer tab #1 (state
  preserved: expanded dirs, selection) and opens a fresh independent
  Explorer tab #2 next to it. Tab #2 is activated. Sidebar collapses.
- Subsequent clicks each open one more independent Explorer tab.
- Each tab carries its own `expanded_dirs` and `selected_path`. Expanding
  `src/` in one tab does not affect any other.
- Closing the last Explorer tab copies its state back into the sidebar,
  re-shows the sidebar at its previous width, and the workspace is back
  to the pre-popout state -- nothing visually jumps.
- Opening files from any Explorer tab opens an editor tab in the usual way.

Architecture:

- New `ExplorerState { expanded_dirs, selected_path, status }` struct in
  `src/gpui_workspace/sidebar.rs`. The three fields used to live as loose
  members of `WorkspacePrototype`; Phase 1 was a pure mechanical refactor
  to bundle them, with no behaviour change.
- `WorkspacePrototype` now owns both `sidebar_explorer: ExplorerState`
  (the parked / sidebar-backed explorer) and
  `explorers: BTreeMap<u64, ExplorerState>` (one entry per Explorer tab,
  keyed by tab id -- same shape as the existing `terminals` and
  `file_editors` maps).
- `WorkspaceSurface::Explorer` enum variant added. Title is `"Explorer"`.
- `active_explorer_state_mut()` helper returns the active tab's state when
  the active surface is Explorer, else falls back to `sidebar_explorer`.
  Click handlers (`toggle_explorer_dir`, `open_sidebar_file`'s selection
  write) route through this helper, so they automatically write to the
  right place without needing to know whether they're being called from
  the sidebar or a tab.
- Tree rendering extracted from `workspace_sidebar` into a shared
  `explorer_tree_panel(id, entries, selected_path, has_project, cx)`
  function. Both the sidebar and the new Explorer pane render through it
  with their own per-instance ids.
- `open_new_explorer_tab` implements the popout: if `explorers` is empty,
  it creates the first tab from a clone of `sidebar_explorer` and hides
  the sidebar; then always creates a fresh second tab. Same dispatch
  shape `open_footer_surface` already uses for Terminal.
- `close_tab` re-dock branch: when the removed tab is an Explorer and the
  map becomes empty, the removed state is moved back into
  `sidebar_explorer` and the sidebar re-shows.
- `open_project` / `close_project` now also call a new
  `close_explorer_tabs()` helper so switching projects can't leave stale
  Explorer tabs pointing at the old root.

Phasing kept commits and review surface small, and let me run `cargo
check` between phases without ever leaving the tree in a non-compiling
state. Phases 1 and 2 (refactor + data model) were invisible to users;
Phase 3 wired the render path; Phase 4 was the user-visible payoff;
Phase 5 was the project-switch fix and a full-build verification.

Coordination with the parallel terminal agent:

- The other agent's command-palette work landed in `gpui_workspace.rs`
  during Phase 2 (added `MenuShowCommandPalette`,
  `palette: CommandPaletteState` field, cmd-shift-p binding). All their
  edits sat in disjoint regions from mine -- actions! macro, keybindings,
  Render listeners. My edits sat on the surface enum, the explorer
  fields, `focus_surface`, `close_tab`, `open_*` functions, and footer.
  No textual conflicts at any phase.

Files touched across the feature:

- `src/gpui_workspace.rs` -- enum variant, fields, helper, `open_new_
  explorer_tab`, footer routing, re-dock branch in `close_tab`.
- `src/gpui_workspace/sidebar.rs` -- `ExplorerState`, `explorer_tree_
  panel`.
- `src/gpui_workspace/project.rs` -- field migration (~30 mechanical

- `src/gpui_workspace/panes.rs` -- `Explorer` render arm, `explorers`

- `src/gpui_workspace/footer.rs` -- Explorer button.

Deliberately deferred (flagged in roadmap, low risk to defer):

  Sidebar still has it; tabs would need the menu state routed to the

- `cmd-shift-E` keybinding for "new explorer tab." Footer button works.

  can manually re-show the sidebar without closing tabs, which loosens

## Terminal font picker + Display/Monospace layout toggle

Building on this morning's per-cell shaping, added user-facing font
selection plus a layout-mode toggle. The terminal can now render
proportional fonts (Atkinson Hyperlegible, Helvetica, Georgia, Palatino,
Verdana) with natural flow when **Display** mode is selected, while
keeping the strict monospace grid as the default.

User flow:

- Appearances → Terminal now has a **Layout** row (Monospace | Display)
  above a single **Font** row that filters to fonts appropriate for the
  active mode. Monospace shows Default / Menlo / Courier. Display shows
  the five proportional families. A muted caveat under the Font row in
  Display mode warns that TUIs and box drawing will look broken.
- Each font writes through the existing `font_family: Option<String>`
  config field; the terminal's `terminal_font()` already honored it.
- The Layout toggle drives a new `terminal_layout: TerminalLayoutMode`
  enum on `Config`. Switching modes auto-clears `font_family` when the
  active selection doesn't belong to the new mode's font list, so the
  picker is never showing a hidden selection.

Renderer (gpui_terminal.rs):

- Branched the prepaint loop on `terminal_config.terminal_layout`.
  Monospace path is unchanged (per-cell shaping at `col * advance`).
  Display path calls a new `terminal_row_flow` helper that shapes the
  whole row through one `shape_line` call and returns a
  `RowLayout { shaped, col_offsets }` where `col_offsets[col]` is the
  x-pixel where column `col` starts, derived from `ShapedLine::x_for_index`.
- New Display-mode rect helpers replace the alacritty helpers that
  assume monospace `advance`:
  - `display_mode_selection_rects` reads cell ranges from a new
    `Terminal::selection_cells()` returning `(viewport_row, col_start,
    col_end)` and maps them through the per-row offset table.
  - `display_mode_background_rects` coalesces adjacent same-bg cells
    per row and uses offsets for x/width.
  - `display_mode_decoration_rects` walks per-cell decoration flags
    (UNDERLINE, DOUBLE_UNDERLINE, UNDERCURL, DOTTED, DASHED, STRIKEOUT)
    and builds rects at each cell's actual shaped extent.
- `cursor_quad` now accepts an optional `&[f32]` of column offsets. In
  Display mode, the cursor's x and width come from `offsets[col]` and
  `offsets[col + 1] - offsets[col]`, so a block cursor over a `W` is
  naturally wider than over an `i`.
- Per-row offset tables are cached on `TerminalSurface::last_row_offsets`
  in `paint`, so `point_to_grid` can binary-search them via
  `col_for_local_x` to map clicks back to grid columns without dividing
  by `advance`. Falls back to monospace advance math when no cache exists.

Out of scope for this pass: alacritty still wraps by column count
(assumes monospace), so in Display mode a long line whose visual width
exceeds the pane will overflow the right edge. Reflow-by-pixel-width is
a much larger change to alacritty's reflow logic and not worth blocking
on for v1.

Files touched:

- `src/config/model.rs` -- `TerminalLayoutMode` enum + `terminal_layout`
  field on `Config`, defaulting to `Monospace`.
- `src/config.rs` -- public re-export of the enum.
- `src/gpui_workspace/appearance_actions.rs` -- `set_terminal_font_family`
  and `set_terminal_layout_mode` actions; the latter clears stranded
  font selections on mode switch.
- `src/gpui_workspace/appearances.rs` -- replaced the dual-row font UI
  with a single Layout toggle + mode-filtered Font row + conditional
  caveat text. Curated font constants exposed as `pub(super)`.
- `src/gpui_terminal/text.rs` -- `RowLayout` struct + `terminal_row_flow`
  helper that shapes a whole row and emits per-column x-offsets, with
  TextRun coalescing across adjacent same-style cells.
- `src/gpui_terminal.rs` -- prepaint branch on layout mode; Display-mode
  rect helpers (`display_mode_selection_rects`,
  `display_mode_background_rects`, `display_mode_decoration_rects`);
  `cursor_quad` signature extended; `last_row_offsets` cache on
  TerminalSurface; `point_to_grid` honors offsets when present.
- `src/terminal/selection.rs` -- `selection_cells()` helper returning
  `(viewport_row, col_start, col_end)` ranges, mirroring the structure
  inside `selection_rects` but skipping the pixel multiplication.

Verification:

- `cargo check --bin gpui-workspace` clean.
- Live test: Monospace mode renders identically to baseline; Display
  mode flows Atkinson / Helvetica / Georgia / Palatino / Verdana with

  on; click hit-testing lands on the correct visual character; selection
  rectangles match the shaped extents.

Mono picker was trimmed at end of session to just Default / Menlo /
Courier -- SF Mono and JetBrains Mono looked off against the monospace
baseline. Display set kept at the full five families.

## IME / preedit handling (roadmap item 5)

The editor and terminal both had `EntityInputHandler` impls but treated
every composition step as a final insert. The visible bug was that each
preedit got appended instead of replacing the previous preedit -- CJK
input came out duplicated, and dictation produced fragments. Fixed both
surfaces by tracking marked-text state through the AppKit
`setMarkedText:selectedRange:replacementRange:` contract.

Editor changes (`src/gpui_editor.rs`, `src/gpui_editor/input.rs`):

- New `marked_range: Option<(Position, Position)>` field on
  `EditorPrototype`.
- `edit_active` clears `marked_range` whenever a buffer mutation
  happens, so non-IME paths (typing, paste, undo, etc.) drop stale
  composition state. IME paths re-set it after `edit_active` returns.
- `marked_text_range` returns the UTF-16 range of the active
  composition.
- `unmark_text` clears it.
- `replace_text_in_range` -- when called with `range_utf16 = None` (the
  OS commit path), targets the marked range first, falling back to the
  cursor selection. Clears `marked_range` after committing.
- `replace_and_mark_text_in_range` -- target = explicit range, else
  previous marked range, else current selection (so subsequent
  preedits overwrite the previous preedit). Applies
  `new_selected_range_utf16` to position the cursor inside the just-
  inserted preedit. Sets `marked_range = (start, start + len(new_text))`
  after `edit_active`. Empty `new_text` cancels the composition.
- New helper `compute_text_end_pos(start, text)` for position math
  without a buffer ref.

Terminal changes (`src/gpui_terminal.rs`):

- New `ime_preedit: String` buffered locally so intermediate composition
  steps are NOT forwarded to the PTY; only the final committed text
  reaches the shell. Previously each composition step was sent as bytes,
  mangling CJK / Hangul input.
- `replace_and_mark_text_in_range` writes to `ime_preedit` (does not
  touch the PTY).
- `replace_text_in_range` (commit) clears the preedit and writes the
  resolved text via the existing `write_bytes` / `write_paste` paths.
- `unmark_text` clears the preedit (composition cancelled -- no bytes
  sent).
- `marked_text_range`, `selected_text_range`, `text_for_range` now
  report based on the buffered preedit so the OS can position its
  composition popover correctly.
- Imports `stacker::utf16::{char_index_to_utf16_index,
  utf16_index_to_char_index}` (same helper the stacker already uses).

Tests:

- Three new unit tests for `compute_text_end_pos`: single line,
  newlines, CJK. The stacker's `set_marked_text` has prior coverage
  and provided the design reference for both surfaces.

Deferred: visible underline rendering for marked text on both surfaces.
macOS shows its own composition popover, so the user-facing feedback is
correct; the underline is the polish-not-correctness part of IME.

## Pane partitioning -- Phase 1 (roadmap item 6, menu-driven)

The existing "Join Tabs" mechanism in `tab_groups::TabGroup` rendered
every pair as a fixed left/right `flex` row. The review called it "a
side-by-side hack, not a real split tree." First pass at real splits:
add an orientation axis, render horizontal vs vertical from the same
data structure, and expose user-facing "Partition" actions in the macOS
menu bar that spawn a fresh terminal in the new region rather than
marrying two existing tabs.

User flow:

- macOS top menu **Tab > Partition > Partition Vertical** -- splits the
  active shelf into two side-by-side shelves with a vertical divider.
  The active tab stays on the left as `primary` (focus preserved); a
  fresh terminal lands on the right.
- **Tab > Partition > Partition Horizontal** -- splits into top/bottom
  shelves with a horizontal divider. New terminal lands on the bottom.
- Drag the divider to resize: left/right for vertical, up/down for
  horizontal.
- If the active tab is already inside a partition, both menu items
  silently no-op. Use **Separate Tabs** first.

Model changes (`src/tab_groups/model.rs`):

- New `PartitionAxis { Vertical, Horizontal }` enum, default
  `Vertical`. `Vertical` means a vertical divider line between two
  side-by-side shelves; `Horizontal` means a horizontal divider line
  between two stacked shelves. The naming matches the visible divider.
- `TabGroup` gains an `axis` field; `TabGroup::new` takes axis.
- New `TabGroupState::join_pair_with_axis(primary, secondary, axis)`.
  The old `join_pair` is a thin wrapper defaulting to `Vertical`, so
  existing callers are unchanged.
- Re-exported `PartitionAxis` from `src/tab_groups/mod.rs`.

Tab-manager changes (`src/gpui_tabs.rs`):

- `GpuiJoinedTabs` carries `axis`.
- `GpuiTabManager::join_pair_with_axis` mirrors the model layer.

Workspace changes (`src/gpui_workspace.rs`):

- `JoinedWorkspacePanes` carries `axis`.
- `joined_panes_for_active` reads it through from the group.
- New action types `MenuPartitionVertical` / `MenuPartitionHorizontal`.
- New `partition_active_with_new_terminal(axis, window, cx)`: allocates
  a fresh terminal tab and calls `join_pair_with_axis(active,
  new_terminal, axis)`; active tab stays focused as `primary`.
  Defensive rollback if the pairing fails (it only can if primary ==
  secondary, which can't happen here).
- `Tab > Partition > Vertical / Horizontal` submenu added to the
  existing `cx.set_menus` block.
- `on_action` listeners wired for both new actions.

Menu-action handlers (`src/gpui_workspace/menu_actions.rs`):

- `menu_partition_vertical` / `menu_partition_horizontal` delegate to
  `partition_active_with_new_terminal`.

Rendering changes (`src/gpui_workspace/panes.rs`):

- `JoinedPaneResizeDrag` now carries `axis` so its drag preview rotates
  (vertical bar 2x40 vs horizontal bar 40x2).
- The joined container uses `flex` row for `Vertical` and `flex_col()`
  for `Horizontal`.
- Drag-math switches between `(position.x - left) / width` and

## Shader pipeline -- M1 + M2

Replaced the nine-rectangle "smoke" stand-in with a real WGSL fragment
shader running through a wgpu render pipeline, composited under the
terminal via GPUI's `Window::paint_surface` hook. Two milestones from
`daily-growth/roadmaps/shaders-05-14-2026.md` landed in one session.

### M1 -- shader bridge harness

The architectural ground truth had to be established first: GPUI 0.2.2's
`Scene::Primitive` enum is closed, so the only public path to put
arbitrary per-pixel content under text on macOS is
`Window::paint_surface(bounds, image_buffer: CVPixelBuffer)`. M1's job
was just to prove that hook end-to-end before any shader work.

- Added `src/effects/` module: `host.rs` (CVPixelBuffer allocator + the
  bridge), `element.rs` (GPUI `Element` wrapper that self-schedules via
  `request_animation_frame` for continuous animation), `shaders/`
  directory for WGSL files.
- First attempt used `kCVPixelFormatType_32BGRA` for the pixel format;
  the blade renderer panicked immediately with `assert_eq!` on the
  format. GPUI 0.2.2 hard-codes the surface assertion to
  `kCVPixelFormatType_420YpCbCr8BiPlanarFullRange` (`'420f'`) -- NV12
  biplanar YUV. The `paint_surface` hook was designed for video-frame
  consumers, not arbitrary RGB output. Rewrote the buffer fill to
  produce NV12: Y plane at full resolution, CbCr interleaved at half
  resolution, BT.601 full-range conversion at the boundary.
- Added IOSurface + Metal-compat attributes via `CVPixelBufferKeys` so
  the buffer is GPU-resident (no CPU copy when GPUI samples it).
- Wired a temporary "Shader Bridge (M1)" test box on the Terminal
  appearance page that drove the element from a time-varying CPU
  gradient. Confirmed live: rainbow gradient + sweeping diagonal band
  visible.

### M2 -- real WGSL smoke shader, workspace-wide

Dispatched two parallel agents (research-only WGSL design + isolated-
worktree wgpu plumbing), integrated the results, then consolidated UX.

- `src/effects/shaders/smoke.wgsl` -- domain-warped FBM, 4 octaves of
  quintic-Hermite value noise, Quilez "flow noise" trick (warp's time
  drives the inner warp coordinates), 3-stop palette mapping, gamma-
  biased toward dark so text reads cleanly on top. ~80 lines of WGSL.
  Bandlimit floor of 32 px (smallest octave wavelength) -- no shimmer
  under bilinear sampling.
- `src/effects/host.rs` -- wgpu Instance / Device / Queue / 64-byte
  uniform buffer / bind group / render pipeline built once in
  `EffectsHost::new`. `render_frame(time, w, h, EffectParams)` allocates
  a per-frame render texture + 256-aligned staging buffer, encodes a
  fullscreen-triangle render pass, copies texture-to-buffer, blocks on
  `map_async` via `pollster`, then writes the staging RGBA into the
  NV12 CVPixelBuffer via the existing BT.601 conversion. Readback path
  instead of zero-copy IOSurface import for v1 -- bridging `metal-rs`'s
  Texture to `objc2-metal`'s MTLTexture protocol would have been a
  cross-FFI transmute we'd rather defer.
- `EffectsHost::with_shared` -- thread-local singleton accessed via
  closure (the inner `CFDictionary` is `!Sync` so a real `static
  OnceLock` doesn't work). Without this, every Appearances repaint
  spun up a fresh wgpu device.
- `app_time_seconds()` -- a global animation clock so all
  `EffectsElement` instances share the same `time` uniform and a newly-
  created element doesn't snap back to t=0.
- `EffectsElement::with_intensity` / `with_palette` builders thread
  config-driven values from the Appearances UI through to the shader.
- `terminal_shader_effect_layer(&Config)` in `gpui_terminal.rs` returns
  `Some(div)` containing a full-bleed `EffectsElement` only when
  `effects.background == "smoke"` and `effects.enabled`. Mounted at
  two sites in `panes.rs`: alongside `terminal_background_layer` in
  both the shared joined-terminal container path and the solo-terminal
  workspace_surface_pane path. Two side-by-side joined terminals share
  one continuous shader field; solo terminals get their own.
- Retired the `"smoke"` arm in
  `gpui_terminal/effects.rs::terminal_background_effects` -- the
  nine-rectangle code is no longer reachable in smoke mode.
- Terminal transparency: two opaque background fills in
  `TerminalSurface::render` had to be gated. The `terminal_body` `bg`
  fill (line 698) and the `root` `bg` fill (line 767) both needed
  `&& !uses_shader_effect`, otherwise the terminal's own paint covered
  the shader layer entirely. Without this fix the smoke rendered into
  the buffer correctly but was painted over -- the user surfaced this
  during M2 verification.
- Five palette presets in the Terminal Appearances tab: Mauve, Aqua,
  Ember, Forest, Slate. Each sets the three palette stops the shader
  reads. `Smoke Intensity` slider drives the shader's intensity uniform
  live; the small preview rendered alongside uses the same EffectsHost,
  so what you see in the panel is exactly what shows behind the terminal.
- Placement: palette + intensity + preview live directly under the
  Background row in the Terminal tab (not at the bottom, not on a
  dedicated Effects tab -- both were attempted, both moved at user
  request).
