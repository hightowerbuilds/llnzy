# 05-14-2026 Summary

## Critical review of the codebase

Walked the tree from the perspective of a senior dev who has shipped IDEs and
terminal emulators, looking at what is built, what is missing, and where the
foundations are load-bearing versus fragile.

What is built (and built well):

- A hand-written, tree-sitter-highlighted, rope-backed editor with undo,
  multi-cursor scaffolding, snippet parser, git gutter, project search, and an
  in-house LSP client speaking 11 servers.
- A real PTY terminal on alacritty_terminal with scrollback, OSC handling,
  selection, and a joined-pane background workflow.
- Discipline markers: ~590 tests, zero TODOs in the tree, structured
  `error_log.rs`, panic-hook-to-crash-log in `main.rs`, SIGPIPE ignored,
  `#[profile.dev]` tuned for typing latency.

Structural concerns surfaced:

- Monolithic surfaces at the size cliff (`gpui_editor.rs` 1,350 lines,
  `gpui_workspace.rs` 1,077, `gpui_terminal.rs` 1,002, `gpui_sketch.rs` 1,204,
  `stacker/cli.rs` 1,410). No plugin boundary, no extension API, no scripting
  surface, no IPC contract between surfaces.
- Polling-not-eventing in `gpui_terminal.rs:37` -- `POLL_INTERVAL_MS = 16`
  wakes 60x/sec to drain the PTY whether or not anything happened.
- Fixed `CELL_WIDTH: f32 = 9.0` decoupled from font, font size, and DPI; wide
  characters (CJK, emoji) will desynchronize rows.
- `snippet_support: Some(false)` in `lsp/client/init.rs:35` despite the
  snippet parser already living in `editor/snippet.rs`.
- Vim mode is a stub on the view model with no operator-pending state machine.

Conspicuously missing for a "developer workspace": command palette, fuzzy file
finder, real pane splits, minimap/breadcrumbs/outline, code-folding UI,
debugger/DAP, remote dev, extension surface, project-wide refactor preview UI,
terminal split, IME/preedit handling, accessibility, auto-update,
signing/notarization, and a Linux/Windows CI matrix.

Code-level smells: 393 `unwrap`/`expect`/`panic!` sites, `gpui = "=0.2.2"`
exact-pinned with no upgrade plan, `tree-sitter-toml` commented out, 14 MB
`LLNZY.dmg` checked into git, no fuzzing or property tests, no benchmarks, no
`cargo deny`/`cargo audit` in CI.

The product question: there is no single thesis tying terminal + editor +
sketch + prompt manager + appearances together. Zed has "fast collaborative."
Helix has "modal, no config." Cursor has "AI native." LLNZY currently reads as
"the things Luke uses, in one window" -- a great reason to build it and a
difficult reason to ship it to anyone else.

## Terminal cell metrics + wide-char handling (roadmap item 1)

Replaced the hardcoded `CELL_WIDTH = 9.0` / `LINE_HEIGHT = 20.0` pair in
`gpui_terminal.rs` with a `CellMetrics` value computed each frame from the
actual terminal font and the configured `terminal.line_height` multiplier.

Changes:

- Introduced a `CellMetrics { advance, line_height }` struct, replacing the two
  module constants. The constants are retained only as `FALLBACK_CELL_WIDTH` /
  `FALLBACK_LINE_HEIGHT` for the first frame before the font resolves.
- Added `compute_cell_metrics(window, base_font, font_size, config)` that
  asks `TextSystem::em_advance` for the natural monospace advance of the
  resolved font and combines `font_size * config.line_height` for the
  vertical step.
- Wired the metrics through every geometry site in `gpui_terminal.rs`:
  `point_to_grid`, `bounds_for_range`, `terminal_grid_size`, `cursor_quad`,
  `selection_rects`, `background_rects`, `decoration_rects`,
  `url_decoration_rects`, the scroll-wheel handler, and the per-line paint
  origin.
- Cached the metrics on `TerminalSurface` as `last_metrics` so event-handler
  paths (mouse hit-testing, IME bounds) use the same numbers as the painted
  geometry. The cache updates inside `resize_for_bounds`, which is called
  from `paint` immediately after `prepaint` produces the metrics.
- Updated `gpui_terminal/effects.rs` so `terminal_cursor_effects` takes
  metrics and drops its `CELL_WIDTH` / `LINE_HEIGHT` imports.
- Updated `gpui_terminal/text.rs` so `terminal_row_runs` skips cells flagged
  `WIDE_CHAR_SPACER` or `LEADING_WIDE_CHAR_SPACER`. The wide glyph itself
  sits in the preceding cell and its natural advance carries it across the
  spacer column; previously the spacer was emitted as a space, pushing every
  cell after a CJK character one column to the right.

Tests:

- Added `terminal::tests::wide_char_occupies_two_cells_with_spacer_flag` to
  pin alacritty's wide-char layout (`中` lands at column 0 with
  `WIDE_CHAR`, column 1 carries `WIDE_CHAR_SPACER`, `A` lands at column 2).

## Live smoke test surfaced two deeper structural issues

The first live launch immediately exposed two problems the unit tests could
not catch.

### Cell advance was measured against a different font than the renderer used

Initial fix attempted: call `TextSystem::em_advance(resolve_font(base_font),
font_size)` and use the result as the cell width.

The cursor still drifted ~10 columns past the end of typed input. Reason:
`em_advance` and the actual `shape_text` rendering of the row do not always
resolve to the same font when the requested family is missing. Berkeley Mono
(the previous default) is not installed on this Mac, so `em_advance`
measured one fallback and `shape_text` rendered with another. Two different
advance widths, accumulating drift across the row.

Second fix: replaced `em_advance` with a `shape_line` probe of `"MMMMMMMMMM"`
through the exact same shaping pipeline that paints rows, dividing the total
shaped width by 10. Same code path means same font resolution by
construction. Also disabled `calt` (programming ligatures) on the terminal
font via `gpui::FontFeatures::disable_ligatures()` because sequences like
`=>`, `->`, `==`, and `!=` collapse into single glyphs in many monospace
fonts and break grid alignment regardless of how the cell width is measured.

The probe-based measurement removed about 80% of the drift but the cursor
was still visibly ahead of typed text. Time for the structural fix.

### The alacritty -> GPUI bridge handed whole rows to `shape_text`

Root cause: every row of the terminal was being concatenated into a single
`String` and shaped through `TextSystem::shape_text`, which is GPUI's
proportional text layout path. Even with a "monospace" font and ligatures
disabled, the shaper can apply per-glyph kerning, fallback metrics,
subpixel rounding, and OpenType features beyond `calt` -- and any tiny
per-glyph drift accumulates across an 80-column row. The cursor was anchored
to `col * advance` while the rendered text was anchored to "wherever the
shaper put each glyph." They could not stay aligned in principle, only by
luck. The user separately reported ASCII art and TUI box drawing (Claude
Code, Gemini) rendering with broken corners and misaligned edges -- same
root cause: the row-wide shape call can't preserve a strict per-column
grid.

Structural fix: replaced row-wide shaping with **per-cell shaping**.

- Removed `terminal_row_runs` (returned one `String` + style runs per row).
- Added `terminal_row_glyphs` that returns a `Vec<TerminalCellGlyph>`,
  one entry per visible cell with its grid column, character, and
  styled `TextRun`. Empty space cells with default style are skipped
  entirely (backgrounds are painted via `Terminal::background_rects`).
- In `prepaint`, loop the glyphs and shape each through `shape_line`,
  storing the resulting `ShapedLine` together with its `(row, col)`.
- Changed `TerminalPaintLine` to carry both `row` and `col`, and switched
  from `WrappedLine` to `ShapedLine` (the simpler shape result that
  matches single-glyph shaping).
- In `paint`, anchor every glyph at exactly `(TERMINAL_PADDING + col *
  metrics.advance, TERMINAL_PADDING + row * metrics.line_height)`. The
  grid alignment is now structural: cursor at column K and glyph at
  column K share the same x coordinate by definition, not by shaper
  cooperation.

With per-cell shaping in, alignment was exact -- but the second live test
showed every character sitting in a cell roughly twice its glyph width.
The grid was correct; the cells were too wide.

### The default font family didn't exist on this Mac

`gpui_terminal.rs` set `.font_family("Berkeley Mono")` on the wrapping div.
Berkeley Mono is a commercial font and was not installed. GPUI resolved the
family through its fallback stack to SF Pro -- a proportional font. The
probe `"MMMMMMMMMM"` measured the width of ten capital M's in SF Pro
(generous), so the cell width was too wide for the average rendered char,
producing visible inter-character gaps. The alignment was perfect; the
spacing looked wrong.

Fix: changed the default `.font_family("Berkeley Mono")` to
`.font_family("Menlo")`. Menlo is a true monospace and ships with every
macOS install, so font resolution does not fall back. Probe and renderer
both produce identical, consistent advances. Users can still override via
`terminal.font_family` in config; Berkeley Mono will work for them if it
is installed.

## Event-driven PTY render loop (roadmap item 2)

Replaced the 60 Hz poll loop on the terminal render task with an event-
driven path. The PTY reader thread was already blocking on `reader.read`
at the OS level, so the wasted work lived entirely in the GPUI side: a
16 ms timer waking the task every frame just to call `try_read` on an
empty channel.

Changes:

- `src/pty.rs` -- added `wakeup: Arc<tokio::sync::Notify>` to `Pty`. The
  reader thread now calls `notify_one()` after every read outcome (data,
  EOF, error path) so the render task wakes on real events and never
  misses a disconnect.
- `src/pty.rs` -- added `Pty::wakeup_handle()` that clones the notifier
  out for consumers.
- `src/session.rs` -- forwarded the notifier with `Session::wakeup_handle`.
- `src/gpui_terminal.rs` -- replaced `start_poll_task` (16 ms timer) with
  `start_event_task`. Each iteration:
  1. Re-fetches the live session's wakeup handle via `surface.update`.
     The re-fetch is required because Cmd-R restart swaps in a fresh
     `Pty` with a fresh notifier.
  2. Awaits `Notify::notified` with a 500 ms safety-net timeout via
     `gpui::FutureExt::with_timeout`. The timeout exists only to cover
     the rare window where the session is replaced between fetch and
     await; under normal operation the task sleeps until real output.
  3. Calls `process_output` on the surface. Returns `cx.notify()` so the
     repaint only fires when data actually changed.
- Removed the old `POLL_INTERVAL_MS = 16` constant; the surviving
  `IDLE_FALLBACK_MS = 500` is just a recovery heartbeat.

Idle CPU sampled from `top -stats pid,command,cpu -pid <workspace>`
dropped to 0.0% from the prior baseline that kept the task ticking at
60 Hz regardless of activity. Live test reported as "feels excellent."

Verification:

```sh
cargo check --all-targets
cargo clippy --all-targets   # no warnings in pty.rs / session.rs / gpui_terminal.rs
cargo test --lib pty
cargo test --test pty_roundtrip
cargo test --test terminal_emulation
cargo test --lib -- --skip platform::packaging   # 533 passed
top -l 1 -stats pid,command,cpu -pid <workspace>  # 0.0% at idle
```

## Multi-explorer tabs feature (separate roadmap)

In parallel with the terminal-side work, scoped and built a new feature:
multiple file explorers as workspace tabs, mirroring how terminals already
work. Roadmap saved at `daily-growth/roadmaps/future/explorer-tabs.md`,
implemented end-to-end in five phases on `main` alongside the other
agent's terminal changes.

User-visible behaviour:

- New **Explorer** button in the footer between Home and Terminal.
- First click pops the sidebar's explorer out as Explorer tab #1 (state
  preserved: expanded dirs, selection) and opens a fresh independent
  Explorer tab #2 next to it. Tab #2 is activated. Sidebar collapses.
- Subsequent clicks each open one more independent Explorer tab.
- Each tab carries its own `expanded_dirs` and `selected_path`. Expanding
  `src/` in one tab does not affect any other.
- Closing the last Explorer tab copies its state back into the sidebar,
  re-shows the sidebar at its previous width, and the workspace is back
  to the pre-popout state -- nothing visually jumps.
- Opening files from any Explorer tab opens an editor tab in the usual way.

Architecture:

- New `ExplorerState { expanded_dirs, selected_path, status }` struct in
  `src/gpui_workspace/sidebar.rs`. The three fields used to live as loose
  members of `WorkspacePrototype`; Phase 1 was a pure mechanical refactor
  to bundle them, with no behaviour change.
- `WorkspacePrototype` now owns both `sidebar_explorer: ExplorerState`
  (the parked / sidebar-backed explorer) and
  `explorers: BTreeMap<u64, ExplorerState>` (one entry per Explorer tab,
  keyed by tab id -- same shape as the existing `terminals` and
  `file_editors` maps).
- `WorkspaceSurface::Explorer` enum variant added. Title is `"Explorer"`.
- `active_explorer_state_mut()` helper returns the active tab's state when
  the active surface is Explorer, else falls back to `sidebar_explorer`.
  Click handlers (`toggle_explorer_dir`, `open_sidebar_file`'s selection
  write) route through this helper, so they automatically write to the
  right place without needing to know whether they're being called from
  the sidebar or a tab.
- Tree rendering extracted from `workspace_sidebar` into a shared
  `explorer_tree_panel(id, entries, selected_path, has_project, cx)`
  function. Both the sidebar and the new Explorer pane render through it
  with their own per-instance ids.
- `open_new_explorer_tab` implements the popout: if `explorers` is empty,
  it creates the first tab from a clone of `sidebar_explorer` and hides
  the sidebar; then always creates a fresh second tab. Same dispatch
  shape `open_footer_surface` already uses for Terminal.
- `close_tab` re-dock branch: when the removed tab is an Explorer and the
  map becomes empty, the removed state is moved back into
  `sidebar_explorer` and the sidebar re-shows.
- `open_project` / `close_project` now also call a new
  `close_explorer_tabs()` helper so switching projects can't leave stale
  Explorer tabs pointing at the old root.

Phasing kept commits and review surface small, and let me run `cargo
check` between phases without ever leaving the tree in a non-compiling
state. Phases 1 and 2 (refactor + data model) were invisible to users;
Phase 3 wired the render path; Phase 4 was the user-visible payoff;
Phase 5 was the project-switch fix and a full-build verification.

Coordination with the parallel terminal agent:

- The other agent's command-palette work landed in `gpui_workspace.rs`
  during Phase 2 (added `MenuShowCommandPalette`,
  `palette: CommandPaletteState` field, cmd-shift-p binding). All their
  edits sat in disjoint regions from mine -- actions! macro, keybindings,
  Render listeners. My edits sat on the surface enum, the explorer
  fields, `focus_surface`, `close_tab`, `open_*` functions, and footer.
  No textual conflicts at any phase.

Files touched across the feature:

- `src/gpui_workspace.rs` -- enum variant, fields, helper, `open_new_
  explorer_tab`, footer routing, re-dock branch in `close_tab`.
- `src/gpui_workspace/sidebar.rs` -- `ExplorerState`, `explorer_tree_
  panel`.
- `src/gpui_workspace/project.rs` -- field migration (~30 mechanical
  sites), `close_explorer_tabs`, click-handler routing through helper.
- `src/gpui_workspace/panes.rs` -- `Explorer` render arm, `explorers`

- `src/gpui_workspace/footer.rs` -- Explorer button.

Deliberately deferred (flagged in roadmap, low risk to defer):

- Right-click context menu (rename / delete / move) inside Explorer tabs.
  Sidebar still has it; tabs would need the menu state routed to the
  active tab.
- `cmd-shift-E` keybinding for "new explorer tab." Footer button works.
- Sidebar bumper toggle while Explorer tabs exist -- currently the user
  can manually re-show the sidebar without closing tabs, which loosens
  the "either sidebar or tabs, never both" invariant. Not broken, just
  worth a real-use pass before deciding to lock down.
- Explicit two-Explorer-tab join via `GpuiTabManager::join_pair`. Should
  work for free since the tab-management layer is surface-agnostic;
  needs eyes-on-the-window confirmation.
