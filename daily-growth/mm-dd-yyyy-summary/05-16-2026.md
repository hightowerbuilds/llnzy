## Error log noise triage and fixes

- Diagnosed the 1006-warning error log under
  `daily-growth/error-log-challenge/error-log-superlist.md`. Original hypothesis
  (mouse / shaders / buffer overload) ruled out. Two unrelated root causes
  identified:
  - ~1000 of the 1006 warnings: workspace recovery snapshot serialization
    failing because `WorkspaceRecoverySnapshot.tab_name_overrides` was a
    `BTreeMap<u64, String>`, which TOML cannot encode (non-string map key).
    Every periodic persist call after a tab rename emitted the same warning
    and silently dropped the snapshot.
  - 6 of the 1006 warnings: tree-sitter highlight queries for C and bash
    referenced node types (`_Thread_local`, `-eq`) that the current grammar
    crate versions do not expose, so both queries failed to compile and those
    two languages were silently dropped from syntax highlighting.

- Recovery snapshot refactor in `src/gpui_workspace/recovery.rs`:
  - Added `WorkspaceRecoveryTabNameOverride { id, name }` and changed the
    on-disk snapshot field from `BTreeMap<u64, String>` to a
    `Vec<WorkspaceRecoveryTabNameOverride>`. In-memory
    `WorkspaceRecoveryPlan.tab_name_overrides` stays as `BTreeMap<u64, String>`
    so call sites do not change shape.
  - Updated `plan_restore` to convert the vector back into the in-memory
    `BTreeMap` while preserving existing filters (drop empty names, drop ids
    that no longer match a valid tab).
  - Updated the snapshot constructor in `src/gpui_workspace.rs` to build the
    vector representation from the in-memory `BTreeMap`.
  - Added regression test `save_load_round_trips_tab_name_overrides` that
    serializes overrides to disk, reloads, and confirms `plan_restore` rebuilds
    the original id-to-name mapping. This test would have failed loudly the
    first time the persist path met a renamed tab.

- Tree-sitter highlight query repair in `src/editor/syntax.rs`:
  - Added a `every_builtin_highlight_query_compiles` test that compiles every
    bundled highlight source against its grammar, so the editor's silent
    `log::warn` path can no longer hide future drift.
  - Removed `_Atomic`, `_Noreturn`, and `_Thread_local` from the C keyword
    list. The current `tree-sitter-c` grammar does not expose these as
    separate node types. The C highlight query now compiles and the rest of C
    highlighting (comments, strings, types, calls, preprocessor) is active
    again.
  - Removed the test-condition flags (`-eq`, `-ne`, `-lt`, `-gt`, `-le`,
    `-ge`, `-z`, `-n`, `-f`, `-d`, `-e`, `-r`, `-w`, `-x`) from the bash
    operator list. The current `tree-sitter-bash` grammar treats these as
    plain string content inside test expressions, not as terminal operator
    nodes. Bash highlighting (comments, strings, variables, command names,
    keywords, structural operators) is active again.

## Performance and durability roadmap â€” Phase A (render path cleanup)

- Created `daily-growth/roadmaps/performance-and-durability-roadmap-05-16-2026.md`
  capturing a four-phase audit-driven plan (render path, GPU gating,
  durability, small wins). Goal: improve behavior on low-end macOS (8GB
  Intel, base M1 Air) without changing any user-visible behavior.

- **A1**. `src/gpui_workspace.rs` â€” `persist_workspace_recovery` is now
  throttled to at most once per 5 seconds for non-clean-shutdown calls.
  Clean-shutdown calls (Quit path) bypass the throttle and persist
  immediately. New constant `RECOVERY_PERSIST_INTERVAL` and field
  `last_recovery_persist_attempt: Option<Instant>`. Existing cached-equality
  short-circuit retained as a second guard. Removes per-frame snapshot
  construction and the disk-write attempt from the render thread.

- **A2**. Workspace `Render::render` no longer calls `load_workspace_queue`
  from `src/gpui_workspace/footer.rs`. Added
  `StackerPrototype::queued_prompts()` accessor; the workspace now reads the
  Stacker entity's in-memory queue directly. The Stacker's own 1 s
  refresh tick continues to pick up external `llnzy stacker` CLI
  mutations. Removed the now-unused `load_workspace_queue` helper.

- **A3**. Workspace `Render::render` no longer rebuilds the explorer entry
  list every frame. New helper `WorkspacePrototype::explorer_entries`
  caches the entries against a `(workspace_root, expanded_dirs)`
  signature; render only rebuilds when that signature changes. Local
  sidebar mutations (expand/collapse, rename, workspace root change) all
  flow through `expanded_dirs` so they invalidate the cache naturally.
  Trade-off: external file changes from outside LLNZY now refresh on
  next user interaction rather than incidentally via the 60 Hz render
  loop. This is closer to most editors' file-tree behavior and matches
  the explicit-refresh model used elsewhere in the app.

- **A4**. `EditorAppearanceConfig.syntax_colors` and
  `EditorAppearance.syntax_colors` are now
  `Arc<FxHashMap<HighlightGroup, [u8; 3]>>` instead of `FxHashMap<...>`.
  `for_language` now does an `Arc::clone` (refcount bump) instead of a
  full HashMap clone. Read sites continue to use `&appearance.syntax_colors`
  unchanged thanks to deref coercion.

- **A5**. `TerminalSurface.config` is now `Arc<Config>` instead of an owned
  `Config`. `set_config` takes `Arc<Config>` and stores the Arc directly.
  In `apply_appearance_config`, the workspace wraps its appearance Config
  in a single Arc and shares it across all terminals via `Arc::clone`,
  replacing the previous per-terminal Config clone. With multiple joined
  terminals this eliminates duplicate Config bodies (each holds a full
  set of theme/syntax/effects fields).
