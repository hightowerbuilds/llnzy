# 05-12-2026 Summary

## GPUI workspace/editor focus

Started the day by re-orienting on the codebase after the GPUI migration. The
current project shape is:

- The legacy production app still lives on the `winit + wgpu + egui` path.
- The new GPUI workbench is feature-gated behind `gpui-workspace`.
- GPUI now has surfaces for Stacker, editor, terminal, sketch, Appearances, and
  shared workspace chrome.
- The immediate problem is the GPUI code editor: files open, but mouse
  interaction in the editor is effectively dead.

The important user report: clicking a code file opens it, but the mouse cannot
meaningfully interact with the code editor. Click-to-place-cursor, drag
selection, and scroll behavior are not working from the user's perspective.

## Investigation

Read the GPUI editor and workspace path:

- `src/gpui_workspace.rs`
- `src/gpui_editor.rs`
- related GPUI examples from the pinned `gpui = 0.2.2` crate

The editor already had code intended to support mouse placement, drag
selection, and scroll. The questionable part was where that handling lived:

- `EditorInputElement` registered raw `window.on_mouse_event(...)` listeners
  during paint.
- That hidden/custom input element also stored `last_text_bounds`, which
  `position_for_point` uses to map mouse points to `(line, col)`.
- GPUI's own text-input example uses normal hit-tested element listeners on the
  visible input container, with the custom input element only providing text
  input / IME geometry.

That made the first working theory: the GPUI migration left editor mouse
handling attached to the wrong layer. If the custom hidden element's bounds,
paint order, or event registration are off, the visible editor can render while
all useful mouse events miss the editor model.

## Attempted patch

Changed `src/gpui_editor.rs` so the visible editor body owns normal GPUI
mouse handling:

- Added `on_editor_mouse_down` as a direct wrapper around the existing
  point-based mouse handler.
- Attached these listeners to the visible editor body:
  - `on_mouse_down`
  - `on_mouse_up`
  - `on_mouse_up_out`
  - `on_mouse_move`
  - `on_scroll_wheel`
- Anchored the custom input layer with `.top_0().left_0()`.
- Removed the custom input element's raw paint-time mouse listeners, leaving it
  responsible for `Window::handle_input` and text-input bounds only.

Verification:

```sh
cargo check --features gpui-workspace --bin gpui-workspace
cargo check --features gpui-editor --bin gpui-editor
```

Both passed.

## Current state

The patch compiles, but it should **not** be treated as a confirmed fix. The
user reported that the attempted change produced no useful effect, and this is
starting to resemble an earlier loop where changes compiled but did not affect
the running behavior.

Current likely explanations to verify before more code changes:

- The running app may not be the binary/path being edited (`gpui-workspace` vs
  legacy app vs stale bundle).
- The active editor surface may not be the `EditorPrototype` instance expected.
- A parent GPUI layer may still be intercepting or occluding editor hitboxes.
- `last_text_bounds` may still not correspond to the visible code area.
- The sidebar/open-file flow may be showing a different editor state than the
  one receiving focus/events.
