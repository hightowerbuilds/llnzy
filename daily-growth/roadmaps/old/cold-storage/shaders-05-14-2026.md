# LLNZY Shader Pipeline Roadmap - 05-14-2026

Goal: replace the nine-rectangle "smoke" stand-in (`src/gpui_terminal/effects.rs:123-143`) and the static aurora bands with real WGSL fragment shaders running on the GPU, rendered once-per-frame into an IOSurface-backed CVPixelBuffer and submitted to GPUI via `Window::paint_surface`. Same hook supports four more effects: fire, rain (window droplets), wind canopy, and a fifth "user shaders" slot fed from `~/.llnzy/shaders/`.

The thesis: llnzy is sitting on top of one of the best graphics stacks in the industry and treating it like a flat-rectangle compositor. The user experience should feel like a Metal demo running underneath the terminal, not a CSS gradient.

## The Constraint and the Escape Hatch

GPUI 0.2.2's `Scene::Primitive` enum is closed - no `paint_with_renderer`, no `Element` extension for raw render passes. The only public hook into the underlying GPU on macOS is:

```rust
Window::paint_surface(bounds: Bounds<Pixels>, image_buffer: CVPixelBuffer)
```

GPUI samples the CVPixelBuffer as a Metal texture in its own renderer. A CVPixelBuffer with the `kCVPixelBufferIOSurfacePropertiesKey` attribute is GPU memory backed by an IOSurface - the same physical allocation our render pipeline writes into. Zero copies, zero CPU round-trip.

That hook is what we build the entire effect system around.

## Decision: wgpu (with naga, Metal-only features)

The 05-14-2026 research session compared wgpu against direct `objc2-metal`. wgpu wins, decisively, on one criterion: **shader-author UX**. Users authoring `~/.llnzy/shaders/smoke.wgsl` get span-aware diagnostics from naga ("line 14, column 3: expected expression, found ';'"). The `objc2-metal` path returns `MTLDevice.newLibraryWithSource` errors as opaque `NSError` strings. For a product that exposes shader authoring, that gap is fatal.

The tradeoffs we accept:

- **Binary cost**: ~6-10 MB added in release with `wgpu = { default-features = false, features = ["metal"] }`. llnzy already ships at ~45 MB; this is noise.
- **HAL instability**: CVPixelBuffer import goes through `wgpu_hal::api::Metal`, an explicitly unstable crate. Surface area is ~30 lines in one module; easy to update on minor wgpu bumps.
- **Apple-only for now**: the Metal feature gate keeps the door open to enable `vulkan` / `dx12` / `webgl2` later without rewriting.

Existing dep already in tree: `notify = "7"` for file watching (`Cargo.toml:44`).
Adds required: `wgpu`, `wgpu-hal`, `naga`, `metal` (transitively pulled by wgpu-hal's Metal backend), `objc2-core-video`, `objc2-core-foundation`, `notify-debouncer-mini`.

## Architecture

Five distinct subsystems. Live under a new top-level module `src/effects/` (the existing `src/gpui_terminal/effects.rs` becomes a thin shim that delegates to the new module for the smoke/aurora cases and stays put for CRT/particle-trail decoration that is pre-textured paint quads).

### `effects::host` - the shader host

Owns:

- `wgpu::Instance`, `wgpu::Device`, `wgpu::Queue` - created once at app start.
- `CVMetalTextureCache` (via `objc2-core-video`) - reused across frames.
- A `CVPixelBufferPool` sized for backing-pixel resolution. Backing pixels not logical pixels - `window.scale_factor() * logical_size`. This is the fix for the bilinear-double-sample shimmer Bart Wronski's analysis flagged.
- The current `ShaderPipeline` slot (and a "next pipeline" channel for hot-reload swap).
- The frame loop entry point: `render_frame(time, size) -> CVPixelBuffer`.

The render loop per frame:

1. Pull a CVPixelBuffer from the pool.
2. Wrap it as `MTLTexture` via the texture cache (zero-copy when IOSurface-backed).
3. Import as a `wgpu::Texture` via `device.create_texture_from_hal::<wgpu_hal::api::Metal>`.
4. Begin a render pass with that texture as the color target.
5. Bind the active `ShaderPipeline` and a uniform buffer with `(time, resolution, scale, custom_uniforms)`.
6. Draw 3 vertices - the canonical fullscreen-triangle trick, no vertex buffer needed.
7. Submit, return the CVPixelBuffer to the caller.

Roughly 250-350 lines for the whole module.

### `effects::pipeline` - WGSL compile + validate (one-shot at startup)

Hot-reload is **explicitly deferred** for v1 (05-14-2026 product call). Shaders are compiled once at app start; editing a shader requires a restart to see the change.

For each shader file at startup:

1. **Read** the WGSL source (builtins via `include_str!`; user shaders via `std::fs::read_to_string`).
2. **Validate** via `naga::front::wgsl::parse_str()` -> `naga::valid::Validator::validate()`. On builtin failure, that's a compile-time bug we should catch in CI (write a test that loads every builtin and validates). On user-shader failure, log the diagnostic to the existing `error_log.rs` and skip that shader from the selectable list.
3. **Build** the wgpu pipeline via `Device::create_shader_module` + `create_render_pipeline`. Stored in a `HashMap<EffectId, RenderPipeline>` on the host.

That's it. No file watcher, no background thread, no swap channel, no toast. ~80 lines instead of ~250.

Hot-reload can come back as a discrete future roadmap item once the effect set is mature - the architecture doesn't preclude it, we just don't pay for it now.

### `effects::workspace_layer` - the paint_surface bridge

The workspace-wide background layer that today is `crate::gpui_terminal::terminal_background_layer` (used for image backgrounds). Replace with a unified `effects_layer` element that:

- Sits above the editor surface but below every terminal pane.
- Calls `effects::host::render_frame(time, bounds.size)` each `prepaint`.
- Calls `window.paint_surface(bounds, cv_pixel_buffer)` in `paint`.
- Two terminal panes side-by-side see the same continuous shader output because there is only one layer beneath both.

Same architectural slot as image backgrounds. The "smoke is per-terminal-tile, not contiguous" problem dissolves by construction.

### `effects::config` - mode + uniforms

`Config.effects.background` becomes a richer enum:

```rust
pub enum BackgroundEffect {
    None,
    Image(BackgroundImage),
    Builtin(BuiltinEffect),  // Smoke, Aurora, Fire, Rain, Canopy
    UserShader { path: PathBuf, uniforms: UniformOverrides },
}
```

Each builtin's WGSL is embedded via `include_str!` at compile time and registered in a default `~/.llnzy/shaders/` layout on first run. Users can edit the built-ins (the file watcher picks them up) or drop new files in.

### `effects::shadertoy_compat` - small but worth doing

A thin WGSL preamble injected into user shaders that provides Shadertoy-style globals: `iTime`, `iResolution`, `iMouse`. Lets users paste Shadertoy WGSL with one or two find-replace edits. Free goodwill.

## Effect Catalog

Each effect is a `.wgsl` file in `src/effects/shaders/` + a uniform schema. Sketches below; the actual shader code lands as the milestone for that effect.

### 1. Smoke (built-in default)

**Technique**: Domain-warped FBM. The Inigo Quilez formula `color = fbm(p + fbm(p + fbm(p + t*v)))` with 4 octaves of 2D gradient noise per FBM call. 12 noise evals per fragment; ~0.6 ms at 1440p backing on M1.

**Uniforms**: `time`, `resolution`, `intensity` (0..1, controls alpha), `color1`, `color2`, `color3` (palette).

**Configurable in the Appearances panel**: intensity, color palette. Hot-reload-editable shader source.

### 2. Fire

**Technique**: same FBM primitive, but UV translated upward by `time * speed`, then sampled into a warm color ramp keyed by row position (hotter at the bottom, smoke-toned at the top). Multiply by a tall vertical mask so the fire is bounded.

**Uniforms**: `time`, `resolution`, `intensity`, `flame_height` (mask cutoff), `palette_warm`/`palette_smoke`.

**Reference**: Shadertoy `Mtf3D7`, `lsf3RH`.

### 3. Aurora

**Technique**: Two stacked horizontal noise bands at different scales, each with their own scrolling speed, multiplied by a column-position mask that's softer in the middle. Add a green->magenta hue ramp keyed by FBM-noise output.

**Uniforms**: `time`, `resolution`, `intensity`, `band_count` (1..4), `palette_aurora`.

**Reference**: any of half a dozen Shadertoy aurora examples; ours can be visually distinctive by picking a non-obvious palette.

### 4. Trees (windy canopy)

**Technique**: Procedural leaf-shape stamps in a fragment shader. For each fragment, hash its position into a "leaf cluster" cell, render a leaf SDF inside the cell, and offset the leaf's center by `sin(time * wind_freq + cluster_hash) * wind_strength` so each leaf bobs on its own phase. Stack 3-4 octaves of leaf clusters at decreasing sizes for canopy depth. Color ramp from deep green (back layer) to bright green (front layer).

**Uniforms**: `time`, `resolution`, `intensity`, `wind_strength`, `wind_direction`, `palette_canopy`.

**Newer territory** than the other four. Worth a small standalone test page in the Appearances panel before integrating.

### 5. Rain (window droplets)

**Technique**: This one is genuinely harder than the rest because *true* refraction needs to read from the underlying scene texture, which GPUI does not give us. Two paths:

- **v1 (achievable now)**: Droplet shader operates over its own gaussian-blurred FBM backdrop. The blur creates a "what's behind the window" approximation. Droplet positions are spawned via a Poisson-disc point distribution evolving over time, each droplet has a 2D distortion kernel that magnifies/displaces the backdrop UVs underneath it. Looks like rain on a window of frosted glass.
- **v2 (deferred)**: True content refraction. Requires a path to render the terminal/editor content to an offscreen texture first, then sample it from the droplet shader. Architectural change to GPUI's draw order - not blocked, but a separate roadmap item.

Land v1 in this pass. v2 goes in `daily-growth/roadmaps/future/`.

## Landing Order and Milestones

Five milestones. Each lands behind a Cargo feature flag (`shaders`) until milestone 4, so a botched intermediate state never blocks the main build.

### M1 - Harness only (no visible change)

- Add wgpu/naga/objc2-core-video deps with `default-features = false, features = ["metal"]`.
- Build `effects::host` with the CVPixelBuffer pool, the wgpu-hal import path, the fullscreen-triangle pipeline.
- Hardcoded "shader" that just clears to a solid color. Wire it into a new test surface in the Appearances panel ("Effects > Test Pattern") so we can see the bridge working end-to-end before touching the actual background.
- Verify: open the test surface, confirm a solid color rectangle is drawn via paint_surface and updates each frame.
- Expected size: ~400 lines of new code. **This is the load-bearing milestone**. If wgpu-hal IOSurface import doesn't behave on real hardware, we find out here before we've sunk effort into shaders.

### M2 - Smoke (real) + the Effects tab

- Write `src/effects/shaders/smoke.wgsl`. Domain-warped FBM, 30-50 lines of WGSL.
- Wire `effects::workspace_layer` and route `BackgroundEffect::Builtin(Smoke)` through it.
- Retire the nine-rectangle smoke from `src/gpui_terminal/effects.rs:123-143`. Aurora and CRT stay for now.
- **Add a new `Effects` tab to the Appearances panel** (next to Theme/Terminal/Editor/Markdown/Sketch). The tab lists each available effect with a thumbnail and per-effect controls. M2 ships the tab with one effect (smoke) and its controls: intensity slider, 3-color palette.
- Verify: two terminal panes side-by-side, smoke is one continuous field across both. Cursor/text reads cleanly over it. Effects tab's intensity slider drives the shader uniform live.

### M3 - Fire and Aurora

- `src/effects/shaders/fire.wgsl` and `aurora.wgsl`.
- Retire the rectangle-based aurora from `effects.rs`.
- All controls land in the new **Effects** tab (M2 introduced it). Each effect's tab section: thumbnail preview, intensity slider, palette swatches, and any effect-specific uniforms (fire: flame height; aurora: band count).

### M4 - Trees and Rain

- `src/effects/shaders/canopy.wgsl` (windy canopy procedural).
- `src/effects/shaders/rain.wgsl` (v1, frosted-glass backdrop droplets).
- Effects tab gains controls: canopy gets wind strength + direction, rain gets droplet density.
- Drop the `shaders` feature flag; the new pipeline is now the default.

Each milestone is mergeable on its own; M1 alone is shippable as a no-visible-change refactor that proves the rendering bridge.

## User-Authored Shaders

First-class from M3 forward, not a "phase N" deferral. The `~/.llnzy/shaders/` directory layout:

```
~/.llnzy/shaders/
  smoke.wgsl
  aurora.wgsl
  fire.wgsl
  canopy.wgsl
  rain.wgsl
  custom/                       # user-authored, picked up automatically
    luke-thunderstorm.wgsl
    luke-glitch-grid.wgsl
```

The contract for a user shader, documented in a README in the same directory:

- WGSL only.
- One entry point `@fragment fn fs_main(@builtin(position) coord: vec4<f32>) -> @location(0) vec4<f32>`.
- A standard uniform block at `@group(0) @binding(0)` exposing `time: f32`, `resolution: vec2<f32>`, `scale: f32`, plus a 4x `vec4<f32>` "user params" slot the Appearances panel can drive with sliders.
- Output color in linear sRGB, premultiplied alpha optional.

Shadertoy compatibility shim: a small WGSL preamble we prepend if the file begins with `// shadertoy` provides `iTime`, `iResolution`, `iMouse` aliases. Lowers the friction for users porting from the web.

In the Appearances panel a "Custom" tab lists every `.wgsl` file under `~/.llnzy/shaders/custom/` with a thumbnail (the rendered effect at 128x128, cached). Click to activate. Edit in any text editor; hot-reload picks it up.

## Out of Scope for v1

- **True content refraction for rain v2**: deferred to a separate roadmap. Requires offscreen-rendering the terminal content first, which is its own architectural change to draw order.
- **Particles**: removed from the effect catalog entirely per the 05-14 product call.
- **Compute shaders / simulation state across frames**: for v1 every effect is stateless per-fragment-per-frame. Frame-to-frame state (e.g., advecting smoke through curl noise, evolving droplet trails) is achievable in the same pipeline with a second pingpong CVPixelBuffer; deferred to a future "stateful effects" pass once the stateless five are solid.
- **Linux / Windows**: wgpu's feature gate accommodates it later. Today llnzy is macOS-only and the IOSurface path is macOS-specific.
- **Thermal management**: idle laptops on battery should probably get a reduced-effects path (lower octave count, lower frame rate). Worth doing eventually; not a v1 blocker.
- **Audio reactivity**: surprisingly tempting but a separate product call. Deferred.

## Open Questions

These need a call before M1 starts:

1. ~~**wgpu version pin**~~ - **Decided 05-14-2026**: take whatever the latest stable wgpu is the day we start M1 and pin exactly. If "latest" turns out to be a pre-release / RC, drop back to the most recent stable. Treat upgrades as discrete tasks - do not track main.
2. ~~**Hot-reload scope**~~ - **Decided 05-14-2026**: no hot-reload for v1. Shaders compile once at startup. Edit-and-restart loop is fine for the launch effect set. Revisit if/when user-authored shaders become a real product surface.
3. ~~**Effect controls in Appearances**~~ - **Decided 05-14-2026**: all mutability lives in the Appearances panel, and gets a dedicated **Effects** tab (next to Theme/Terminal/Editor/Markdown/Sketch) so each effect's controls have room to breathe.
4. **The shimmer-mitigation choice**: render at 2x backing for box-downsample, render at 1x with bandlimited noise, or render at 1x with a 0.10-0.15 temporal blend? My lean: bandlimited noise + temporal blend. Free, kills shimmer, blurs motion a touch which is fine for ambient backgrounds. 2x rendering is the heavy-hammer fallback if shimmer still shows on Retina.

Resolving these unblocks M1.

## Reference

Pinned from the 05-14-2026 research sessions, for the implementer:

- **paint_surface entry point**: `crates/gpui/src/window.rs` line ~3181 in gpui 0.2.2.
- **CVPixelBuffer creation**: `CVPixelBufferCreate` with `kCVPixelBufferMetalCompatibilityKey=true`, `kCVPixelBufferIOSurfacePropertiesKey={}`.
- **Metal texture wrap**: `CVMetalTextureCacheCreateTextureFromImage`. Reuse the cache; don't recreate.
- **wgpu HAL import**: `device.create_texture_from_hal::<wgpu_hal::api::Metal>(hal_tex, &desc)`.
- **Shimmer**: render at `window.scale_factor() * logical_size`, bandlimit noise to ≤ 0.5 / pixel_size (Nyquist), optional `mix(prev, current, 0.15)` temporal blend.
- **WGSL parsing**: `naga::front::wgsl::parse_str()` + `naga::valid::Validator::validate()`. Spans are in the error type. `naga-error` crate (Kneelawk) wraps these into a unified report.
- **Bevy hot-reload pattern**: PR #3031 (RenderPipelineCache), #3966 (asset hot reload). The "build new before drop old" semantic is the load-bearing piece.
- **Shadertoy refs**: smoke (`wttXz8`, `XslGRr`), fire (`Mtf3D7`, `lsf3RH`), aurora (search "aurora borealis shadertoy"), clouds (`3l23Rh`).
- **iquilezles.org**: domain warping (`articles/warp`), gradient noise (`articles/gradientnoise`), FBM (`articles/fbm`). Canonical references for the noise primitives we'll use.
